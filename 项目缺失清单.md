# 项目缺失清单

> 更新依据：`aios/README.md`、`docs/IMPLEMENTATION_PROGRESS.md`、`docs/system-development/19-实现映射与当前进度.md`、`aios/shell/README.md`、`aios/services/updated/README.md`、`aios/services/deviced/README.md`、`aios/providers/README.md`、`aios/compat/README.md`、`aios/observability/README.md`、`.github/workflows/aios-ci.yml`
>
> 目的：基于当前仓库真实实现，重新识别 AIOS 现在“真正还缺什么”。本清单不再把已落地 baseline 误写成空白，而是聚焦“已有基线但尚未闭合的主线缺口”。

## 1. 总体判断

AIOS 当前已经明显超出“只有骨架”的阶段。

仓库里已经存在的，不应再按“几乎没有实现”描述的基线包括：
- 核心服务、本地 IPC、provider / portal / runtime baseline
- bootable image、QEMU bring-up、installer / recovery / updated baseline
- Smithay nested compositor baseline、panel host bridge、chooser / panel / client / smoke 基线
- `deviced` 的 readiness / backend-state / continuous capture / `ui_tree_snapshot` 基线
- first-party provider runtime、registry、health、自注册、恢复 smoke 基线
- compat permission manifest、runtime loader / enforcement、MCP bridge structured contract、bounded local sandbox baseline
- observability schema、validation report、evidence index、release gate report、cross-service correlation / health schema 与 workflow 基线

因此，项目当前最真实的缺失，不是“有没有起步”，而是：

1. 多条主线仍未达到 release-grade
2. 多个能力仍停留在 baseline / partial impl，而不是平台级闭环
3. Tier 1 实机、统一证据、自动阻塞发布门槛仍未完全形成

## 3. 当前真实缺失清单

### A. 平台交付与更新恢复主线仍未闭合

#### A1. first-boot 与 image-level gate 仍未达到发行级
当前缺的是：
- first-boot 幂等、失败留痕、image-level gate 收口
- machine-id / random-seed / installer integration 的发行级验证
- 构建、启动、恢复、更新的一体化 machine-readable 证据

#### A2. update / rollback / recovery 仍缺平台级闭环
当前缺的是：
- `updated` 的真实平台 sysupdate / rollback 闭环
- vendor-specific firmware hook 与 system-specific health probe
- 失败注入、回退、boot-success 的跨重启平台证据

#### A3. Tier 1 实机 bring-up 仍未完成
当前缺的是：
- Tier 1 硬件安装、启动、恢复、更新连续证据
- 真实硬件成功记录，而不只是 QEMU / 本地 restart smoke

### B. 核心控制面仍需从 partial impl 收敛为稳定控制面

#### B1. 契约冻结与共享治理仍未完全收口
当前缺的是：
- contract freeze、兼容性门禁、共享 error code 全面接线
- schema / config loader 在更多服务路径上的统一落地

#### B2. 关键控制面主链仍未闭合
当前缺的是：
- `sessiond` 的 task event query API、semantic memory surface、recovery evidence 收口
- `policyd` 的 approval state machine、audit query、跨服务 policy evidence
- `runtimed` 的 observability export、真实 budget enforcement、backend contract 收敛

#### B3. 持续集成 harness 与失败恢复验证仍不足
当前缺的是：
- 更完整本地 integration harness
- control-plane 级 machine-readable evidence
- 更系统化的跨服务失败恢复验证

### C. Shell / Portal 用户流仍未达到正式交付级别

#### C1. shell / compositor 仍未达到 release-grade
当前缺的是：
- 完整窗口管理、真实 panel 真嵌入、复杂 stacking / role / focus 策略
- DRM/KMS 级完整桌面栈，而不只是 nested baseline

#### C2. chooser / approval / recovery 用户流仍需收敛
当前缺的是：
- formal chooser GUI 的最终收口
- 更完整异常路径、长时稳定性、正式 acceptance artifact
- 与 `sessiond` / `policyd` / provider / recovery 的统一交互体验

#### C3. real screen cast flow 仍未完全闭合
当前缺的是：
- chooser 后真实受控媒体链路
- 与 `deviced` 正式 backend 的主路径打通

### D. Device / Provider / Compat 执行面仍缺正式主路径

#### D1. `deviced` 仍缺真实 backend 与完整 `ui_tree`
当前缺的是：
- 正式 portal / PipeWire / libinput / camera backend
- 完整 AT-SPI live tree、跨桌面支持矩阵与长期稳定性证据
- visible indicator / backend status 的正式 shell 消费面收口

#### D2. provider fleet 与 lifecycle / observability 仍未封板
当前缺的是：
- shell / device / compat provider fleet 扩充
- worker lifecycle、descriptor、health 与统一 observability / audit sink 收口
- 编译级 / 集成级证据的持续化

#### D3. compat / sandbox 仍未达到正式执行级别
当前缺的是：
- browser / office 真桥接实现
- 更正式的 remote auth / provider registration / audit sink
- OS 级 sandbox engine
- 更完整的 compat 与 policy / approval 联动

### E. Runtime backend 与验证治理仍缺最后收口

#### E1. 真实 vendor backend 仍未形成稳定执行链路
当前缺的是：
- vendor GPU / NPU runtime integration
- release-grade runtime backend 执行链路

#### E2. observability / schema / versioning 仍缺工程化收口
当前缺的是：
- schema loader
- runtime validation
- migration strategy
- schema version tests
- 统一 event sink / health exporter 收口

#### E3. 发布治理仍未形成自动阻塞闭环
当前缺的是：
- artifact 命名、retention、failure triage 的统一治理
- regression matrix + owner + command + artifact 对应表
- release checklist 自动阻塞化

## 4. 当前最关键的缺失优先级

按当前项目主线，建议优先级如下：

1. release-grade first-boot / recovery / update / rollback 平台闭环
2. Tier 1 实机 bring-up 与跨重启证据
3. 正式 shell / compositor / chooser / panel 用户流
4. `deviced` 正式 backend 与 real screen cast / capture 主路径
5. 核心控制面 query / approval / audit / budget / harness 收口
6. provider fleet、compat 真桥接与 OS 级 sandbox
7. schema loader、runtime validation、统一 sink / exporter、artifact 治理
8. vendor GPU / NPU runtime backend 集成

## 6. 建议维护方式

建议后续按以下状态维护每条缺口：
- `未开始`
- `骨架完成`
- `部分实现`
- `可运行`
- `已验证`
- `发行级闭环`

并在每次里程碑更新时同步附带：
- 代码路径
- smoke / test 名称
- artifact 路径
- report 路径
- 当前 blocker

这样这份文档才能持续反映“项目真实还缺什么”，而不是停留在过时口径。
