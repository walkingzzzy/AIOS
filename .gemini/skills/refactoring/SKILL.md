# 重构

## 重构原则

> 重构是在**不改变外部行为**的前提下，改善代码内部结构

### 何时重构
- ✅ 添加新功能前
- ✅ 修复 bug 后
- ✅ 代码审查时发现问题
- ✅ 理解代码困难时

### 何时不重构
- ❌ 没有测试覆盖
- ❌ 临近发布期限
- ❌ 代码需要完全重写

## 代码异味（Code Smells）

### 代码层面

| 异味 | 描述 | 解决方案 |
|------|------|---------|
| 重复代码 | 相同逻辑出现多处 | 提取函数/类 |
| 过长函数 | 函数超过50行 | 拆分函数 |
| 过长参数 | 参数超过4个 | 引入参数对象 |
| 过大类 | 类职责过多 | 拆分类 |
| 神秘命名 | 名称无法表达意图 | 重命名 |

### 结构层面

| 异味 | 描述 | 解决方案 |
|------|------|---------|
| 过度耦合 | 模块间依赖过多 | 依赖注入 |
| 特性依恋 | 方法更多使用其他类的数据 | 移动方法 |
| 中间人 | 类只做转发 | 移除中间人 |
| 过度泛化 | 过早抽象 | 内联类/方法 |

## 安全重构步骤

```
1. 确保有测试覆盖
2. 运行测试（确认通过）
3. 进行小步重构
4. 运行测试（确认通过）
5. 提交更改
6. 重复 3-5
```

## 常用重构技术

### 提取函数
```
前：一个长函数做多件事
后：多个小函数各做一件事
```

### 内联函数
```
前：函数只有一行且不增加可读性
后：直接使用函数体
```

### 提取变量
```
前：复杂表达式难以理解
后：用有意义的变量名解释
```

### 引入参数对象
```
前：function(x, y, width, height)
后：function(rectangle)
```

### 移动方法/字段
```
前：方法在错误的类中
后：移动到更合适的类
```

## 性能优化

### 优化时机
```
1. 先让代码正确工作
2. 再让代码清晰可读
3. 最后优化性能（如有必要）
```

### 优化步骤
```
1. 测量（找到瓶颈）
2. 优化（针对瓶颈）
3. 测量（验证优化）
```

### 常见优化
- 缓存重复计算
- 减少 I/O 操作
- 批量处理代替循环
- 使用合适的数据结构

---

## 重构禁忌

- ❌ 在没有测试的情况下重构
- ❌ 一次做太多更改
- ❌ 重构的同时添加功能
- ❌ 忽略编译/测试错误继续重构
