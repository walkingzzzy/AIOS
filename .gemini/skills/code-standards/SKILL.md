# 代码质量标准

## 语言使用强制规范

> ⚠️ **绝对强制使用简体中文**

### 必须使用中文的场景（无例外）

| 场景 | 要求 |
|------|------|
| AI 回复 | ✅ 强制中文 |
| 所有文档 | ✅ 强制中文 |
| 代码注释 | ✅ 强制中文 |
| Git 提交信息 | ✅ 强制中文 |
| 操作日志 | ✅ 强制中文 |
| 审查报告 | ✅ 强制中文 |
| 任务描述 | ✅ 强制中文 |
| 错误提示 | ✅ 强制中文 |
| 测试描述 | ✅ 强制中文 |

**唯一例外**：代码标识符遵循项目既有命名约定

---

## 注释编写规范

- ✅ UTF-8 无 BOM 编码
- ✅ 描述意图、约束与使用方式，不重复代码逻辑
- ❌ 禁止"修改说明"式注释（由版本控制承担）
- ✅ 模块复杂时补充设计理由
- ✅ 简洁明了，直指核心

## 设计原则

### SOLID 原则

| 原则 | 描述 | 要求 |
|------|------|------|
| S - 单一职责 | 一个类只做一件事 | **强制** |
| O - 开闭原则 | 对扩展开放，对修改关闭 | 推荐 |
| L - 里氏替换 | 子类可替换父类 | 推荐 |
| I - 接口隔离 | 接口小而专一 | **强制** |
| D - 依赖倒置 | 依赖抽象，不依赖实现 | **强制** |

### 其他原则

- **DRY**：共享逻辑抽象为复用组件
- **关注点分离**：复杂逻辑先拆分职责再编码
- 禁止临时绑死实现细节

## 实现标准

### 绝对禁止

| 禁止项 | 说明 |
|--------|------|
| ❌ MVP/最小实现 | 提交前必须完成全量功能 |
| ❌ 占位符代码 | 必须完善为完整实现 |
| ❌ 过时/重复代码 | 主动删除保持整洁 |
| ❌ 逃生式代码 | 删除临时绕过 |
| ❌ 向后兼容 | 采用颠覆式破坏性更改 |

### 强制执行

| 强制项 | 说明 |
|--------|------|
| ✅ 语言标准风格 | 符合编程语言规范 |
| ✅ 项目既有风格 | 导入顺序、命名、格式化 |
| ✅ 最佳实践 | 代码质量和可维护性 |
| ✅ 迁移步骤 | 破坏性改动提供迁移/回滚方案 |

---

## 性能意识

- 设计时评估**时间复杂度、内存占用、I/O 影响**
- 识别潜在瓶颈后提供监测或优化建议
- 禁止引入未经评估的昂贵依赖或阻塞操作

---

## 测试思维

- 编码前编制可验证的验收条件
- 对预期失败场景提供处理策略
- 连续三次验证失败必须暂停实现

---

## 简单性定义

```
✅ 每个函数/类只承担单一责任
✅ 禁止过早抽象（重复三次以上再通用化）
✅ 禁止"聪明"技巧，可读性优先
✅ 如需额外解释，说明实现过于复杂
```
