# 项目集成规则

## 架构优先级

> "标准化 + 生态复用"拥有**最高优先级**

| 规则 | 描述 |
|------|------|
| ✅ 优先复用 | 首先查找官方 SDK、社区成熟方案、既有模块 |
| ❌ 禁止自研 | 除非已有实践无法满足且获书面批准 |
| ✅ 删除自研 | 减少维护面，降低技术债务 |
| ✅ 验证兼容 | 引入外部能力时验证与项目标准兼容 |
| ✅ 规划替换 | 对自研实现规划替换或下线时间表 |

---

## 学习代码库

### 必须执行

1. **寻找相似特性**
   - 至少找到 **3 个**相似特性或组件
   - 理解其设计与复用方式

2. **识别通用模式**
   - 识别项目中通用模式与约定
   - 在新实现中**沿用**

3. **优先使用既有工具**
   - 优先使用既有库、工具或辅助函数
   - **避免重复造轮子**

4. **遵循测试编排**
   - 遵循既有测试编排
   - 沿用断言与夹具结构

---

## 工具规范

| 规则 | 描述 |
|------|------|
| ✅ 使用现有构建系统 | 不得私自新增脚本 |
| ✅ 使用既定测试框架 | 沿用现有运行方式 |
| ✅ 使用格式化/静态检查 | 遵循项目设置 |
| ⚠️ 新增工具需论证 | 提供充分论证并获批准 |

---

## 内容唯一性规则

- 每一层级必须自洽掌握自身抽象范围
- **禁止跨层混用**内容
- 必须**引用**其他层的资料而非复制粘贴
- 禁止在高层文档中堆叠实现细节

---

## 重要提醒

### 绝对禁止

- ❌ 在缺乏证据的情况下做出假设
- ❌ 结论不援引现有代码或文档

### 必须做到

- ✅ 复杂任务前完成详尽规划并记录
- ✅ 跨模块或超过5个子任务的工作生成任务分解
- ✅ 复杂任务维护 TODO 清单并更新进度
- ✅ 开发前校验规划文档得到确认
- ✅ 小步交付，确保每次提交可用
- ✅ 同步更新计划文档与进度记录
- ✅ 学习既有实现的优缺点并复用改进
- ✅ 连续三次失败后暂停操作，重新评估
