# 标准工作流程

## 总原则（必须遵循）

- **强制深度思考**：任何时候必须首先梳理问题，这是开发工作的基础
- 不是必要的问题**不要询问用户**，必须自动连续执行
- 问题驱动优先于流程驱动
- 追求充分性而非完整性
- 动态调整而非僵化执行

---

## 标准工作流 6 步骤

| 步骤 | 名称 | 描述 |
|------|------|------|
| 1 | 分析需求 | 理解任务目标和约束 |
| 2 | 获取上下文 | 检索相关代码和文档 |
| 3 | 选择工具 | 确定合适的工具和方法 |
| 4 | 执行任务 | 实施具体工作 |
| 5 | 验证质量 | 运行测试和检查 |
| 6 | 存储知识 | 记录决策和经验 |

---

## 研究-计划-实施 5 阶段

| 阶段 | 内容 | 禁止 |
|------|------|------|
| 1. 研究 | 阅读材料、厘清约束 | **禁止编码** |
| 2. 计划 | 制定详细计划和成功标准 | - |
| 3. 实施 | 根据计划执行，小步提交 | - |
| 4. 验证 | 运行测试，记录结果 | - |
| 5. 提交 | 准备交付文档和迁移方案 | - |

---

## 迭代改进循环

```
Plan → Context → Implement → Test → Refine
  ↑                                    ↓
  └────────────────────────────────────┘
```

### 每次迭代
1. **计划**：明确本次迭代目标
2. **上下文**：收集必要信息
3. **实现**：编写代码
4. **测试**：验证正确性
5. **改进**：回顾并优化

### 迭代原则
- ✅ 每次迭代保持可交付状态
- ✅ 每次只解决一个问题
- ✅ 失败时立即回滚
- ❌ 不要一次尝试太多变更

## 工作流程阶段定义

### 阶段0：需求理解与上下文收集

**快速通道**：简单任务（<30字，单一目标）直接进入上下文收集

**复杂任务**：
1. 分析需求并识别关键疑问
2. **强制检索清单**（7步，编码前必做）
3. **充分性验证**（7项检查，必须全部通过）
4. **生成上下文摘要**（`.claude/context-summary-[任务名].md`）

### 阶段1：任务规划

- 分析上下文摘要
- 制定计划与验收契约
- 定义接口规格、边界条件、性能要求、测试标准

### 阶段2：代码执行

- 直接编码（使用 Read/Edit/Write）
- 实时记录到 `operations-log.md`
- 遇到问题时分析并调整策略

### 阶段3：质量验证

- 深度审查
- 生成评分和建议（→ `.claude/verification-report.md`）
- 决策规则：
  - ≥90分 → 通过
  - 80-89分 → 仔细审阅
  - <80分 → 退回修改

## 文件结构规范

所有任务工作文件写入 `.claude/` 目录：

```
<project>/.claude/
├── context-summary-[任务名].md   ← 上下文摘要
├── operations-log.md             ← 决策和操作记录
└── verification-report.md        ← 验证报告
```

- [ ] 确认日志文件可写
- [ ] 检索相关代码或文档，确认复用路径

## 开发哲学（强制遵循）

- ✅ 渐进式迭代，每次改动可编译
- ✅ 研读既有代码，吸收经验
- ✅ 务实态度，满足真实需求
- ✅ 表达清晰，拒绝炫技
- ✅ 简单方案，避免过度架构
- ✅ 遵循既有风格
