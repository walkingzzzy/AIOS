# 强制上下文检索

## 核心原则

**绝对禁止**：在未完成上下文检索和验证的情况下直接编码。违反者立即终止任务。

---

## 项目知识库（CLAUDE.md）

在项目根目录创建 `CLAUDE.md` 文件作为项目"大脑"：

```markdown
# 项目名称

## 常用命令
- 构建：`npm run build`
- 测试：`npm test`
- 开发：`npm run dev`

## 核心文件
- 入口：src/index.ts
- 配置：src/config/
- 工具：src/utils/

## 代码风格
- 使用 TypeScript strict 模式
- 命名：camelCase 变量，PascalCase 类

## 架构说明
[项目架构概述]
```

## 检索强度分级

| 任务类型 | 条件 | 执行范围 |
|---------|------|---------|
| 简单任务 | 单文件、<50行、无依赖 | 步骤1-3，简化验证 |
| 中等任务 | 多文件、<200行、少量依赖 | 完整7步，标准验证 |
| 复杂任务 | 架构级、>200行、复杂依赖 | 完整7步+增强验证 |

## 编码前7步强制检索

### □ 步骤1：文件名搜索（必须）
```
find_by_name Pattern="关键词" SearchDirectory="项目路径"
```
- 目标：找到5-10个候选文件
- 记录：找到X个相关文件，重点关注 [列出文件路径]

### □ 步骤2：内容搜索（必须）
```
grep_search Query="函数名|类名|关键逻辑" SearchPath="项目路径" MatchPerLine=true
```
- 目标：找到关键实现位置
- 记录：找到X处实现，重点分析 [file:line]

### □ 步骤3：阅读相似实现（必须≥3个）
```
view_file AbsolutePath="相关文件"
```
- 目标：理解实现模式和设计理由
- 关注点：
  - 实现模式（工厂/单例/装饰器等）
  - 可复用组件（函数/类/工具）
  - 需注意事项（性能/安全/边界）

### □ 步骤4：开源实现搜索（通用功能必做）
```
github.search_code query="具体功能" language:"语言"
```
- 目标：学习最佳实践和避免常见陷阱
- 触发条件：通用算法、数据结构、设计模式

### □ 步骤5：官方文档查询（涉及库/框架必做）
```
context7 resolve-library-id libraryName="库名"
context7 get-library-docs context7CompatibleLibraryID="库ID" topic="主题"
```
- 目标：避免错误用法，掌握最佳实践
- 优势：官方文档，token高效，最新版本

### □ 步骤6：测试代码分析（必须）
```
grep_search Query="describe|it|test" Includes=["*.test.*", "*.spec.*"]
```
- 目标：理解测试策略和覆盖标准
- 关注点：测试框架、断言方式、Mock策略

### □ 步骤7：模式提取和分析（必须）
- 目标：生成项目模式清单
- 输出：
  - 项目约定：命名规范、文件组织、导入顺序
  - 可复用组件：[组件路径列表]
  - 技术选型：为什么用这个方案？

## 上下文充分性验证（7项必查）

**必须全部回答"是"且提供具体证据，否则禁止进入编码阶段。**

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 能说出至少3个相似实现的文件路径？ | [file1:line, file2:line, file3:line] |
| 2 | 理解项目中这类功能的实现模式？ | 模式是 [描述]，因为 [理由] |
| 3 | 知道有哪些可复用的工具函数/类？ | [列出函数/类名和路径] |
| 4 | 理解命名约定和代码风格？ | [具体说明] |
| 5 | 知道如何测试这个功能？ | 参考 [测试文件]，策略是 [描述] |
| 6 | 确认没有重复造轮子？ | 检查了 [模块/文件] |
| 7 | 理解功能的依赖和集成点？ | 依赖 [X]，集成点 [Y] |
