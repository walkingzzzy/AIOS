---

## 一、目标与范围

- **核心目标**: 打通流式响应全链路、补齐 LLM 生命周期钩子、建立统一事件流、完善请求准备钩子、可选推进 MCP SDK 迁移与 GUI Agent 能力评估。
- **范围边界**: 不改三层 AI 协调设计、不改变权限模型、不默认开启 ReAct + O-W、不扩大适配器范围。
- **交付原则**: 先接通链路与可观测性，再进行架构性调整。

---

## 二、周期概览

- **总周期**: 6 周（1 周为一个 sprint）
- **节奏**: 4 个主阶段 + 1 个可选阶段

| 阶段 | 周期 | 目标 | 关键交付 |
|:---|:---|:---|:---|
| Phase 1 | W1-W2 | 流式响应全链路接入 | `processStream` + IPC 事件 + 前端联调 |
| Phase 2 | W3 | LLM 生命周期钩子 | Hook 类型与触发点 + 日志/指标 |
| Phase 3 | W4 | 统一事件流系统 | EventStream + 事件订阅与回放 |
| Phase 4 | W5-W6 | MCP SDK 迁移（可选） | SDK 评估与迁移方案 |

---

## 三、里程碑与验收标准

- **M1: 流式全链路可用**: 客户端可收到 `task:stream-chunk` 与 `task:stream-complete`，支持取消与错误回收。
- **M2: LLM 钩子可观测**: LLM 请求/响应/流式分片均可被 Hook 捕获并记录指标。
- **M3: 统一事件流落地**: 事件流与 Hook 对齐，支持订阅与时间窗口过滤。
- **M4: MCP 迁移评估完成**: 输出迁移方案与影响面清单，是否实施由评审决定。

## 五、质量与测试策略

- **必测路径**: 流式响应、Hook 触发、事件订阅、IPC 协议兼容。
- **回归范围**: 任务执行流程、工具调用、现有非流式路径。
- **建议测试**: 补充 E2E 覆盖流式与取消；单测覆盖 HookManager 与 EventStream。

---

## 六、风险与应对

- **流式事件风暴**: 增加事件限流或裁剪策略，避免内存增长。
- **Hook 增加性能开销**: 控制 Hook 超时与并发执行，保持默认轻量。
- **IPC 兼容性**: 保持现有接口不变，仅扩展新事件通道。
