# 五团队并行开发总计划

## 1. 目标

基于最新代码审查结果，将 AIOS 当前“仍未闭合的主线缺口”拆成 5 条可并行推进、边界清晰、低冲突的开发主线。

这里的规划口径不再把项目描述成“大面积空白”，而是明确区分：
- 哪些能力已经有 baseline / smoke / schema / workflow
- 哪些能力仍未达到 release-grade、平台级闭环或实机验证级别

## 2. 拆分原则

1. 按目录划分唯一责任团队，避免多人同时改同一模块
2. 跨团队协作只通过稳定契约进行：RPC、schema、descriptor、artifact、report
3. 优先收敛“已有基线但未封板”的主线，不重复建设平行功能
4. 功能团队负责主链闭环，团队 E 负责验证、证据与发布门槛收口
5. 所有团队都要输出 machine-readable evidence，而不只输出 README 或人工说明

## 3. 五个并行团队

### 团队 A：平台交付与更新恢复组
负责：镜像、启动、安装、first-boot、update / rollback / recovery、Tier 1 bring-up。
计划书：`团队A-平台交付与更新恢复计划书.md`
当前主线：把已有 bootable image、installer / recovery baseline、updated RPC 与 restart smoke，收敛为 release-grade 平台闭环与实机证据。

### 团队 B：核心控制面与安全治理组
负责：`agentd`、`sessiond`、`policyd`、`runtimed` 及共享 IPC / contract / policy 基线。
计划书：`团队B-核心控制面与安全治理计划书.md`
当前主线：把已有核心服务最小闭环，收敛为 contract freeze、query / approval / audit / budget / observability 可持续集成控制面。

### 团队 C：Shell / Portal 交互组
负责：shell、chooser、panel、portal 用户流与壳层体验。
计划书：`团队C-Shell与Portal交互计划书.md`
当前主线：把已有 Smithay nested compositor baseline、panel host bridge、chooser / panel skeleton / smoke，收敛为正式 GUI 壳层与稳定用户流。

### 团队 D：设备 / Provider / Compat 执行组
负责：`deviced`、`providers/`、`compat/`、sandbox 执行面。
计划书：`团队D-设备ProviderCompat执行计划书.md`
当前主线：把已有 device state / readiness、provider registry、compat permission manifest / runtime baseline、sandbox baseline，收敛为真实 backend、正式 worker contract 与结构化执行证据。

### 团队 E：验证 / 观测 / 发布治理组
负责：observability、schema、validation、CI、artifact、release gate。
计划书：`团队E-验证观测与发布治理计划书.md`
当前主线：把已有 schema baseline、validation report、evidence index、release gate report、workflow，收敛为统一 sink / exporter、版本治理、自动阻塞发布门槛。

## 4. 目录所有权矩阵

- 团队 A：`aios/image/`、`aios/hardware/`、`aios/delivery/`、`aios/services/updated/`
- 团队 B：`aios/services/agentd/`、`aios/services/sessiond/`、`aios/services/policyd/`、`aios/services/runtimed/`、`aios/crates/aios-core/`、`aios/crates/aios-contracts/`、`aios/crates/aios-rpc/`、`aios/policy/`
- 团队 C：`aios/shell/`、`aios/crates/aios-portal/`
- 团队 D：`aios/services/deviced/`、`aios/providers/`、`aios/compat/`、`aios/sdk/`
- 团队 E：`aios/observability/`、`tests/`、`.github/workflows/`、发布/验证类根文档

## 5. 跨团队接口规则

1. 团队 B 冻结核心 RPC、error code、token、policy、route、query 契约
2. 团队 C 只消费团队 B / D 冻结后的 session、policy、provider、device 状态接口，不直接改控制面主逻辑
3. 团队 D 只消费团队 B 的 token / route / approval 契约，不直接定义 shell 侧交互语义
4. 团队 A 与团队 B / E 通过 `updated` RPC、health probe、deployment state、validation evidence 协作
5. 团队 E 不承担主功能实现，但负责 schema、report、artifact、release gate 的统一口径

## 6. 同步节奏建议

- 每周一次架构 / 契约评审：只讨论跨团队接口变更
- 每周两次集成窗口：合并 smoke、artifact、schema、validation report 与 release evidence
- 任何新增高影响接口都必须先补 ADR 或 machine-readable schema
- 里程碑完成必须附带脚本、报告或 artifact 证据，避免仅凭口头声明推进状态

## 7. 第一优先级里程碑

1. 形成 release-grade first-boot / recovery / update / rollback 平台闭环，并拿到 Tier 1 实机证据
2. 形成正式 shell / chooser / panel / portal 用户流，并收敛 compositor 与 panel 真嵌入
3. 形成稳定的核心控制面 integration harness，补齐 query / approval / audit / budget / observability 主链
4. 形成 `deviced + provider + compat + sandbox` 的真实执行路径，而不是 probe / skeleton / fallback 主导
5. 形成 schema loader、runtime validation、统一 sink / exporter、artifact 治理与自动阻塞 release gate 体系
