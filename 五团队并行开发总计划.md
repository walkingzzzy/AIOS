# 五团队并行开发总计划

**版本**: 1.0.0  
**创建日期**: 2026-03-23  
**状态**: 执行层根计划

---

## 1. 目的

本文件是 AIOS 项目执行层的**根计划**。  
自 2026-03-13 起，AIOS 的项目管理口径统一按五个团队并行推进，Phase 与 track 仅用于任务拆分参考。

本文件回答以下问题：

1. 五个团队各自负责什么
2. 每个团队当前的主里程碑是什么
3. 各团队之间的依赖与协作关系
4. 当前全局执行优先级

---

## 2. 团队定义与目录所有权

### 团队 A — 平台交付（镜像 / 启动 / 更新 / 硬件）

| 属性 | 内容 |
|------|------|
| **目录所有权** | `aios/image/`、`aios/hardware/`、`aios/delivery/`、`aios/services/updated/` |
| **主要职责** | 镜像构建、启动链、安装流程、系统更新、回滚、恢复、硬件 bring-up |
| **Phase 覆盖** | Phase 1（Image/Boot）、Phase 4（Update/Recovery）、Phase 5（Tier 1 硬件） |

### 团队 B — 核心控制面（服务 / IPC / 安全治理）

| 属性 | 内容 |
|------|------|
| **目录所有权** | `aios/services/agentd/`、`aios/services/sessiond/`、`aios/services/policyd/`、`aios/services/runtimed/`、`aios/crates/aios-core/`、`aios/crates/aios-contracts/`、`aios/crates/aios-rpc/`、`aios/policy/` |
| **主要职责** | 核心控制面服务、共享 IPC / contract、安全治理、runtime backend |
| **Phase 覆盖** | Phase 2（Core Services）、跨阶段 contract / audit / harness 收敛 |

### 团队 C — Shell 与 Portal（用户交互层）

| 属性 | 内容 |
|------|------|

| **主要职责** | 正式 shell compositor、panel GUI、chooser GUI、portal 交互 |
| **Phase 覆盖** | Phase 3（Shell/Portal 用户流）、Phase 4/5（recovery/backend status 用户消费面） |

### 团队 D — 执行生态（Device / Provider / Compat / SDK）

| 属性 | 内容 |
|------|------|
| **目录所有权** | `aios/services/deviced/`、`aios/providers/`、`aios/compat/`、`aios/sdk/` |

| **Phase 覆盖** | Phase 3（Provider/Compat）、Phase 5（Device）、Phase 6（ui_tree） |

### 团队 E — 观测、验证与发布（CI / 证据 / 发布门槛）

| 属性 | 内容 |
|------|------|

## 4. 全局执行优先级

### 4.1 第一优先级（P0 — 阻塞发布）

1. **团队 C**：形成正式 Shell / Compositor GUI 最小可用版
2. **团队 A**：完成两台 Tier 1 实机 bring-up 并产出 sign-off 证据
3. **团队 B**：接入真实推理 backend（llama.cpp / candle）形成端到端执行闭环

### 4.2 第二优先级（P1 — 阻塞功能完整性）

4. **团队 B**：核心服务联调收敛（六服务 systemd 联动 + IPC 互通）
5. **团队 D**：形成 deviced + provider + compat 真实执行路径
6. **团队 E**：CI / 集成测试 / release gate 持续化

### 4.3 第三优先级（P2 — 阻塞产品化）

7. **团队 A**：vendor-specific firmware hook 适配
8. **团队 D**：provider fleet 生态扩展
9. **团队 E**：release notes 生成与发布工程自动化

## 5. 团队间依赖关系

```text
团队 A（平台交付）
    ↕ 镜像 / delivery / 硬件验证
团队 B（核心控制面）
    ↕ IPC contract / token / audit
团队 C（Shell）──→ 依赖团队 B 提供 RPC 接口
    ↕ portal / chooser / panel 嵌入
团队 D（执行生态）──→ 依赖团队 B 提供 provider registry / policy
    ↕ device / provider / compat
团队 E（验证与发布）──→ 消费所有团队产物进行验证
```

### 5.1 关键跨团队依赖

| 依赖方 | 被依赖方 | 依赖内容 | 状态 |
|--------|---------|---------|------|
| 团队 C | 团队 B | sessiond / policyd / runtimed RPC 接口 | 已满足（skeleton 已有） |
| 团队 C | 团队 B | agentd provider registry lifecycle RPC | 已满足 |
| 团队 D | 团队 B | policyd 执行 token 签发与验证 | 已满足 |
| 团队 D | 团队 B | agentd provider registry 加载链 | 已满足 |
| 团队 A | 团队 B | delivery bundle 中包含核心服务二进制 | 已满足 |
| 团队 E | 全团队 | 各团队 smoke / 证据产出 | 持续依赖 |
| 团队 A | 团队 C | 实机 Shell 启动验证需要 compositor | **阻塞中** |
| 团队 B | 团队 D | runtime-local-inference-provider 真实执行 | **阻塞中** |

## 6. 关键阻塞项（当前 Top 5）

1. **Shell 正式 GUI 几乎为零**（团队 C）：用户无法与系统交互，阻塞所有用户流验证
2. **Tier 1 实机 bring-up 零成功记录**（团队 A）：无法证明系统可在真实硬件运行
3. **真实推理 worker 缺失**（团队 B）：AI 推理能力为空壳，无法执行实际意图
4. **真实 device backend 缺失**（团队 D）：多模态感知无法工作
5. **集成测试覆盖不足**（团队 E）：缺 registry / provider / image 自动化联调

## 8. 管理机制

### 8.1 每周状态同步

- 每周按 `docs/system-development/26-每周状态同步模板.md` 产出同步报告
- 每个团队必须汇报：本周完成项、当前阻塞项、新增 ADR 需求、下周目标

### 8.2 任务管理

- Machine-readable 任务源：`aios/tasks/master-task-list.yaml`、`aios/tasks/execution-board.yaml`
- 完整任务清单：`docs/system-development/21-完整任务清单.md`
- 执行看板：`docs/system-development/22-执行任务看板.md`

### 8.3 完成定义（DoD）

一个任务从 `In Progress` 进入 `Done` 前必须同时满足：

1. 有仓库产物
2. 有最小运行或验证证据
3. 有回退 / 风险说明
4. 有文档映射

### 8.4 口径一致性原则

根计划（本文件）、团队计划、任务清单、执行看板与 machine-readable YAML 必须保持口径一致。任何变更需同步更新所有层级文档，避免口径漂移。
