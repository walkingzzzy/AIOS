# 07 · 信任边界与威胁模型

**版本**: 1.1.0  
**更新日期**: 2026-03-08  
**状态**: P0 必备规格

---

## 1. 目标

本文件定义 AIOS 的安全边界，回答四件事：

1. 谁可以被信任到什么程度
2. 哪些系统对象必须被保护
3. 哪些攻击路径必须在 v1 即被阻断
4. 本地执行、代码沙箱、可信云卸载之间如何划线

## 2. 保护对象

AIOS 必须保护的核心资产包括：

- 用户文件、凭据、令牌、密钥材料
- capability policy、审批状态、执行 token
- 会话上下文、任务历史、审计日志
- 系统配置、更新元数据、恢复入口
- 设备控制能力：摄像头、麦克风、屏幕、网络、蓝牙、USB、输入设备
- 模型上下文、提示词、工具调用记录、外部 provider 凭据
- KV cache、embedding index、语义记忆、工作记忆 scratchpad
- 远端卸载 attestation 信息与路由决策记录

## 3. 主体分类

| 主体 | 信任等级 | 说明 |
|------|----------|------|
| Human User | 高 | 最终责任主体，审批链拥有者 |
| Linux Kernel / Boot Chain | 高 | 真实执行根 |
| Core Services | 高 | 受签名镜像与系统策略保护 |
| `aios-runtimed` | 中高 | 运行时核心，但不是权限豁免源 |
| AI Shell | 中高 | 系统界面，但不直接绕过 policy |
| Local Model Runtime | 中 | 可推理，不天然拥有系统管理权 |
| Code Sandbox | 低到中 | 受限执行环境，只能在沙箱内运行 |
| Remote Model / SaaS | 低 | 默认不可信，只能返回建议或结构化计划 |
| Attested Trusted Cloud | 中 | 受证明、可审计、可禁用的受限远端执行域 |
| Compat Providers | 低到中 | 受限桥接层，默认最小权限 |
| Third-party Workloads | 低 | 必须沙箱隔离 |
| Legacy Console | 低到中 | 过渡期调试工具，不得作为信任根 |

## 4. 信任分区

### Zone 0 · 启动与根信任

- Firmware / Secure Boot
- Bootloader
- Kernel / initrd
- 镜像签名与更新元数据

### Zone 1 · 系统执行基线

- `systemd` 与基础 system units
- `aios-agentd`
- `aios-sessiond`
- `aios-policyd`
- `aios-runtimed`
- `aios-updated`

### Zone 2 · 系统交互与感知层

- AI Shell / compositor
- `aios-deviced`
- 用户会话服务
- 审批 UI / task surface

### Zone 3 · 兼容与受限执行层

- browser / office / MCP / A2A providers
- code sandbox runtime
- legacy UI / CLI
- 第三方应用与外部桥接

### Zone 4 · 远端与非可信输入

- 远端模型输出
- 网页内容、文档内容、邮件内容
- 外部脚本、自动化结果、用户粘贴文本
- 外部 MCP server / external A2A peer

### Zone 5 · 受证明远端执行域

- 通过 attestation 的 trusted cloud offload 节点
- 只允许受控、受审计、可禁用的远端推理或受限处理

## 5. 强制安全原则

### 5.1 remote AI never equals local authority

远端模型、远端 agent、远端 SaaS 的任何输出都只能作为：

- 建议
- 结构化计划
- 待执行参数
- 低信任输入

它们不得天然拥有：

- root 权限
- 直接文件系统写入权
- 直接设备控制权
- 绕过审批的 capability 执行权

### 5.2 LLM is not the trust root

研究文档中的 `LLM-as-Kernel` 不能被误读为“模型就是信任根”。  
AIOS 中真正的信任根仍然是：

- 启动链
- 内核
- system manager
- policy engine
- token 验证链
- 审计链

### 5.3 policy is the only execution gateway

所有高价值操作都必须遵循统一链路：

`intent -> plan -> capability resolution -> policy evaluation -> approval -> execution token -> execution -> audit`

任何绕过 `policyd` 的执行路径都视为架构错误。

### 5.4 tainted content must stay tainted

来自网页、邮件、文档、远端工具、第三方 MCP server 的内容必须保留来源标签与 taint 状态。  
被 taint 的内容不能直接升级为高风险执行依据。

### 5.5 generated code runs only in sandbox

动态代码生成是高价值能力，但只能在受控沙箱中执行，并且：

- 文件系统最小化
- 网络权限显式声明
- 无 root
- 有时间与资源上限
- 输出进入审计链

### 5.6 trusted cloud must be attested and revocable

可信云卸载必须同时满足：

- 节点身份可校验
- 路由可审计
- 处理可禁用
- 默认无状态或等价最小留痕
- 失败时自动回退或拒绝执行

## 6. 主要威胁场景

### T1 · Direct Prompt Injection

风险：用户直接输入恶意指令覆盖系统约束。  
约束：系统提示、能力策略、审批规则不得由自然语言输入直接覆盖。

### T2 · Indirect Prompt Injection

风险：网页、邮件、文档通过内容注入影响 AI 决策。  
约束：所有外部内容必须标注来源；工具调用前必须进行能力级再确认。

### T3 · Over-privileged Compat Provider

风险：兼容层拿到广泛文件系统或设备权限后越权。  
约束：compat 以 capability broker 获取授权，不得直接继承系统主权限。

### T4 · Hallucinated Execution

风险：模型对不存在的资源、路径、用户或设备做错误判断。  
约束：所有执行对象必须在本地解析并验证，执行前落地为结构化参数。

### T5 · Infinite Tool Loop / Agent Loop

风险：工具调用型或 supervisor 型 agent 陷入无意义循环。  
约束：必须有 iteration cap、budget cap、timeout 与人工介入门槛。

### T6 · Runtime Starvation / OOM

风险：高并发推理导致显存耗尽、系统卡死、队头阻塞。  
约束：分级队列、内存预算、背压、拒绝策略必须存在。

### T7 · Approval Bypass

风险：通过 UI 漏洞、API 直连或 legacy 工具绕过审批链。  
约束：审批结果必须由 `policyd` 签发短时 token，执行端只认 token 不认 UI 状态。

### T8 · Malicious MCP / A2A Peer

风险：外部协议对端伪造 schema、返回恶意 payload 或诱导无限联调。  
约束：外部 bridge 默认低信任；schema 校验、超时和来源隔离必须开启。

### T9 · Update Supply Chain Attack

风险：镜像、更新元数据、provider 包被替换。  
约束：签名校验、版本钉死、回滚路径、恢复模式缺一不可。

### T10 · Rogue Trusted Cloud Offload

风险：未证明或已失效的远端节点被错误当作可信节点使用。  
约束：attestation 失败即禁止；高敏感请求默认本地。

## 7. v1 必须具备的安全控制

- `capability` 风险分级
- 高风险能力审批链
- taint source 标记与审计
- 兼容层最小权限模型
- 远端模型输出低信任标记
- 代码沙箱资源限制
- 更新签名校验与失败回滚
- 安全模式 / 恢复模式
- 服务间最小 IPC 白名单
- 迭代次数、超时和资源预算约束

## 8. 暂不要求但建议预留

- TPM 绑定密钥与远程证明
- 完整 MAC 策略集成
- 可验证构建与 SBOM
- 企业策略下发与集中审计
- 多租户或多 seat 高级隔离

## 9. 结论

AIOS 的核心安全结论不是“AI 足够聪明所以可以管系统”，而是：

> **AI 只能参与决策与编排；系统控制权必须由内核、服务管理器、策略引擎、审批链、审计链共同约束。**
