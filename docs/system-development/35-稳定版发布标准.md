# 35 · 稳定版发布标准

**版本**: 1.0.0  
**更新日期**: 2026-03-14  
**状态**: Frozen

---

## 1. 目标

本文定义 AIOS `Stable Release` 的最低标准。

稳定版不等于“功能最多”，而等于：

- 支持边界清晰
- 风险可审计
- 更新可回退
- 硬件与运维证据可复现

## 2. 必须满足的条件

### 2.1 发布治理

- 开发者预览与产品预览标准都已满足
- release gate 的 blocking checks 全部通过
- 硬件证据从“可选”提升为“必需”

### 2.2 平台与硬件

- 至少 2 台 Tier 1 机器完成 bring-up 与恢复证据
- 支持矩阵中的 stable 平台都必须有对应验证记录
- vendor-specific firmware hook 与平台 profile 必须可复用

### 2.3 安全与审计

- 第一轮安全审计已完成并关闭高优先级问题
- 高风险 token / approval / audit 规则进入长期维护基线
- 新增高风险路径必须自动纳入 audit / observability 体系

### 2.4 运维与恢复

- update / rollback / recovery 在声明支持的平台上有稳定证据
- diagnostic bundle 与现场排障路径足以支撑 first-line triage
- 已知限制必须从“工程说明”收敛为“正式支持声明”

### 2.5 兼容性

- 版本兼容矩阵必须覆盖 stable 声明范围
- schema / provider contract / platform profile 的变更必须有 migration note
- 破坏性变更必须有升级路径或明确不兼容声明

## 3. 不允许存在的状态

- 无实机证据却宣称 stable
- 无回退路径却宣称可升级
- 无审计链却宣称高风险能力可生产使用
- 无支持矩阵却宣称“全平台稳定”

## 4. 当前结论

截至 2026-03-14，AIOS 距离 `Stable Release` 仍有明显差距。

主要缺口：

- Tier 1 硬件矩阵尚未完成
- release-grade shell / compositor 尚未完全收口
- 更多 provider / runtime / audit 长期维护约束仍待固化
