# 11 · 应用、兼容层与沙箱模型

**版本**: 1.1.0  
**更新日期**: 2026-03-08  
**状态**: P1/P2 必备规格

---

## 1. 目标

AIOS 不能再把“应用”当作核心抽象，但也不能回避应用与兼容层。  
本文件定义它们在系统中的正确位置，以及动态代码与协议桥接应当如何被限制。

## 2. 对象分类

| 类型 | 定位 | 例子 |
|------|------|------|
| Core Service | 系统本体 | `agentd` `runtimed` `sessiond` `policyd` |
| Shell Component | 系统壳层 | launcher / task surface / notification |
| First-party Provider | 第一方系统扩展 | 文件、网络、通知、系统状态 provider |
| Compat Provider | 旧软件桥接 | browser / office / MCP / A2A bridges |
| Code Sandbox | 受限动态执行环境 | Python / WASM / isolated executor |
| Third-party App | 第三方工作负载 | 编辑器、工具、终端、媒体软件 |
| Automation Fallback | 兜底桥接 | GUI automation / input automation |
| Legacy Console | 过渡期工具 | Electron client / debug CLI |

## 3. 设计原则

### 3.1 system first, app second

- 应用不是 AIOS 的定义核心
- compat 只在原生能力缺失时承担桥接职责
- GUI automation 只能是兜底，不得成为主控制路径

### 3.2 portal / broker first

应用与 compat 不直接拿系统高权，而通过 broker / policy / portal 风格接口申请。

### 3.3 sandbox by default

任何非核心服务对象默认都在沙箱中运行。

### 3.4 code generation is not a free pass

动态代码生成不是越权理由。生成的代码也必须通过受限执行环境、预算和审计链。

## 4. v1 运行模型

### 4.1 Core Services

- 运行在 system units
- 不进入第三方沙箱
- 受最严格代码与依赖治理

### 4.2 Shell Components

- 运行在用户图形会话中
- 通过受控 IPC 访问核心服务
- 不直接拥有 root 或系统配置修改权

### 4.3 Compat Providers

- 以独立 service 或受限 runtime 运行
- 通过 capability broker 请求授权
- 对文件、网络、屏幕、输入、剪贴板采用显式声明
- 必须区分本地 provider 与远端协议桥接 provider

### 4.4 Code Sandbox

推荐用于：

- 解析复杂结构化数据
- 运行受限数据转换脚本
- 执行短生命周期的动态逻辑

必须具备：

- 最小文件系统
- 受控网络
- CPU / memory / timeout 限额
- 审计与结果回传
- 不可持久化高敏感上下文

### 4.5 Third-party Apps

v1 推荐使用**类 Flatpak 的最小权限模型**：

- 文件系统权限尽量缩小
- 优先 portal 而非 blanket access
- 网络默认按声明放开
- 屏幕、录音、摄像头、剪贴板权限独立控制

## 5. 权限基线

### 5.1 文件系统

- 默认仅访问自身数据目录
- 用户文档目录通过显式授权或 portal 打开
- 批量读写用户主目录属于高风险能力

### 5.2 网络

- 第三方 workload 与 compat provider 默认不自动拥有广泛外发能力
- 对外发布、同步、发送类动作属于高风险审批场景
- 代码沙箱默认不开放全网访问

### 5.3 图形与输入

- 屏幕读取、录屏、输入注入、键鼠自动化必须单独授权
- Automation fallback 默认关闭，仅在显式 compat 模式下开启

### 5.4 设备

- 摄像头、麦克风、蓝牙、USB 访问独立声明
- 背景持续占用敏感设备必须有可见状态与撤销入口

### 5.5 模型与上下文

- compat 与第三方 workload 不直接访问系统主记忆
- 需要模型能力时，通过 `runtime.*` 受控调用
- 高敏感上下文默认不暴露给 code sandbox 或远端 bridge

## 6. compat provider 的边界

compat provider 可以做：

- 协议桥接
- 数据格式转换
- 受限的宿主软件交互
- 作为 AIOS 原生能力未完成时的过渡路径

compat provider 不可以做：

- 伪装成系统服务
- 自建 root 级后台守护逻辑
- 直接读写系统敏感目录
- 绕过 `policyd` 或审计链
- 直接决定切换本地 / 远端执行边界

## 7. 协议桥接模型

MCP / A2A 在 AIOS 中的正确位置是：

- 它们是**系统级互操作协议**
- 它们不是权限绕过通道
- 外部对端必须被视作低信任或中低信任实体
- 发现、调用、通知、会话继承都应纳入 policy 与 audit

## 8. 第三方应用模型

AIOS v1 建议把第三方 GUI 软件视为：

- **被托管的工作负载**，而不是系统定义核心
- 可以被启动、聚焦、挂起、通知、授权，但不能反向定义系统能力语义

建议预留对象：

- `workload.bundle`
- `workload.permission`
- `workload.runtime`
- `workload.lifecycle`

这些 workload 对象用于描述被托管工作负载，不再扩展新的 `app.*` 命名空间。  
但它们也不应替代 `system.*` / `runtime.*` / `service.*` / `shell.*` 的主线模型。

## 9. legacy 处置原则

- `packages/client` 继续存在时，统一命名为 `legacy console`
- `packages/daemon` 中可复用逻辑逐步抽取到系统服务
- legacy 工具默认只用于调试、回归测试、过渡演示
- 不得再以 legacy 功能页数量衡量系统成熟度

## 10. 验收标准

### P1

- compat provider 有独立进程边界
- 关键 provider 有最小权限声明
- code sandbox 有最小受限执行环境

### P2

- 文件、网络、屏幕、输入权限可独立治理
- automation fallback 可被显式启停
- 外部协议桥接具备来源隔离与超时控制

### P3

- 第三方 workload 有统一打包 / 运行 / 权限接口
- compat 与原生能力的替换路径明确
- code sandbox 与 compat 审计链统一
