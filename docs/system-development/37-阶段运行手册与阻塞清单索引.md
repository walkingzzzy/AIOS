**版本**: 1.0.0

**状态**: Cross-stage Runbook / Blocker Index

---

## 1. 目标

本文把当前 AIOS 各阶段的运行手册、验证入口与阻塞项收敛到一个索引里，避免后续排障继续分散在多个 README、手册和看板之间。

它回答三个问题：

1. 每个阶段现在以哪份 runbook 或手册为准
2. 当前最影响交付的阻塞项分别是什么
3. 当问题升级时，应先看哪份文档、再补哪类仓库证据

## 2. 使用规则

- 阶段级 runbook 以本文索引为入口，但不替代各子系统的详细设计和专用手册
- 执行看板中的 blocker 变更后，应同步更新本文对应阶段的“当前阻塞”
- 新增高影响恢复/发布动作时，优先补 runbook 或手册，再补看板摘要

## 4. Team A 当前交付阻塞清单

### 4.1 仍可继续推进的非安装项

- 继续维护 Team A 的 update / rollback / recovery observability 与 release gate 对齐
- 保持 Team A 的支持矩阵、兼容矩阵、发布标准、runbook 与仓库实现同步，并维持 `image-build-strategy-report` / validation matrix / CI artifact policy 三者一致
- 收敛 update / recovery 的 machine-readable 审计覆盖与 CI 阻塞规则

### 4.2 当前明确不在本轮处理范围

- 真实安装介质现场操作
- Tier 1 实机 bring-up 执行本身
- vendor-specific firmware hook 的实机接线与厂商工具链落地

## 5. 升级与排障顺序

1. 先看对应阶段的主手册，确认当前标准动作和 artifact 路径
2. 再看 `22-执行任务看板.md`，确认是否已经被列为当前 blocker
3. 若 blocker 影响 release gate，再看 `docs/RELEASE_CHECKLIST.md` 与 `tests/observability/validation-matrix.yaml`
4. 若问题属于高风险能力或恢复路径，额外检查 `tests/observability/high-risk-audit-coverage.yaml`

## 6. 当前结论

AIOS 的 runbook、发布标准、支持矩阵和 operator 手册已经形成第一轮闭环，但阻塞项仍主要集中在：

- 团队 A 的实机安装 / 更新 / 回退连续证据
- 团队 B 向更多真实 nominated machine sign-off 主线扩展 audit evidence export，并持续吸收新增实机 `hardware-validation-evidence.json`
- 团队 C 的 formal shell / chooser / approval GUI 集成
- 团队 D 的正式 native backend 与统一 audit/export
- 团队 E 的实机 evidence blocking gate 与 operator-facing audit evidence export 扩面
