# 08 · 用户、会话与进程模型

**版本**: 1.1.0  
**更新日期**: 2026-03-08  
**状态**: P0 必备规格

---

## 1. 目标

本文件定义 AIOS 的主体与运行时模型，避免继续沿用“client + daemon”式应用思维。

## 2. 用户模型

### 2.1 用户类型

| 类型 | 说明 | 默认权限 |
|------|------|----------|
| `admin` | 设备拥有者 / 管理员 | 可批准高风险系统能力 |
| `standard` | 普通使用者 | 仅可使用个人范围内 capability |
| `operator` | 受限运维角色 | 可执行部分系统维护任务 |
| `service` | 系统服务账户 | 仅用于服务进程，不可交互登录 |
| `guest` | 临时会话 | 无持久敏感状态 |

### 2.2 基本规则

- v1 默认支持单机本地账户
- 企业域账户、远程目录服务不是 v1 阻塞项
- 所有 capability 授权必须绑定用户主体
- AI 不是主体，AI 只能代表某个用户会话发起计划

## 3. 会话模型

### 3.1 会话类型

- `system session`：引导期与核心服务会话
- `user graphical session`：图形桌面 / shell 会话
- `background task session`：离屏任务会话
- `recovery session`：恢复模式下的最小维护会话

### 3.2 v1 会话基线

- 每个用户默认只允许一个主图形会话
- 多图形会话、多 seat 协作不作为 v1 主线
- `sessiond` 负责会话登记、上下文持久化、任务挂接、恢复映射

### 3.3 会话状态机

`created -> active -> suspended -> resumable -> closed -> archived`

要求：

- `active` 会话可发起任务、接收通知、进入审批链
- `suspended` 会话可被恢复，但不能直接继续高风险任务
- `archived` 会话只保留审计与检索价值

## 4. 记忆模型

AIOS 必须把“记忆”作为系统对象，而不是单纯的 prompt 拼接。

### 4.1 Working Memory

- 当前任务 scratchpad
- 当前上下文窗口
- 执行中间态、短期工具返回
- KV cache 引用关系

### 4.2 Episodic Memory

- 会话历史
- 任务历史
- 用户最近操作与反馈轨迹

### 4.3 Semantic Memory

- 长期知识快照
- embedding / index
- 结构化知识条目

### 4.4 Procedural Memory

- policy 模板
- workflow 模板
- tool schema
- 运行时约束与固定操作规范

### 4.5 基本规则

- working memory 默认短生命周期
- episodic / semantic memory 需要明确持久化策略
- procedural memory 需要版本化与审计
- 高敏感上下文禁止无审计地进入远端卸载路径

## 5. 任务与智能体模型

### 5.1 任务对象

任务是 AIOS 的执行单位，最少包含：

- `task_id`
- `session_id`
- `user_id`
- `intent`
- `resolved_capabilities`
- `risk_level`
- `approval_state`
- `execution_state`
- `route_state`
- `audit_refs`

### 5.2 任务生命周期

`draft -> planned -> awaiting_approval -> executing -> succeeded | failed | cancelled -> archived`

规则：

- 任务在 `planned` 后必须固定 capability 集合
- 若 capability 集合变化，必须产生新计划版本
- 审批通过的是计划版本，不是自然语言原文

### 5.3 支持的 agent topology

v1 支持：

- `direct`：单模型直接调用
- `tool-calling`：单智能体 + 工具
- `supervisor`：监督者 + 专长子代理
- `plan-execute`：规划与执行分离

v1 暂不主推：

- `hierarchical teams`

原因：跨团队协调与上下文继承成本过高，容易放大资源与稳定性问题。

## 6. 进程模型

### 6.1 进程类别

| 类别 | 典型组件 | 运行方式 |
|------|----------|----------|
| Core Services | `agentd` `runtimed` `sessiond` `policyd` | system units |
| Device / update services | `deviced` `updated` | system units |
| Shell components | compositor / launcher / notification | user session units |
| Compat providers | browser / office / bridges | restricted service units |
| Model runtimes | local inference backends | dedicated slice / service |
| Code sandbox | Python / WASM / isolated executor | restricted runtime |
| Third-party apps | sandboxed workloads | app runtime / sandbox |
| Legacy tools | Electron console / CLI | debug-only, non-trust-root |

### 6.2 运行规则

- 核心服务不得运行在 Electron 进程内部
- compat provider 不得与核心服务共享同一进程空间
- 本地模型运行时不得自动继承设备或文件全权限
- 调试工具可以读取日志，但不得绕过 `policyd`
- 代码生成型执行必须进入 code sandbox

## 7. 资源模型

AIOS v1 必须基于 `cgroup v2` 建立最小资源隔离。

推荐逻辑切片：

- `system.slice`：基础系统服务
- `aios-core.slice`：`agentd` / `sessiond` / `policyd`
- `aios-runtime.slice`：`runtimed` / local model workers
- `aios-shell.slice`：compositor / shell components
- `aios-device.slice`：capture / perception service
- `aios-compat.slice`：compat providers / code sandboxes
- `app.slice`：第三方工作负载

最小控制项：

- CPU 配额
- 内存上限与 OOM 策略
- IO 权重
- GPU / NPU 使用约束（若有）
- 后台任务优先级
- 队列长度与并发预算

## 8. 进程、会话与记忆的关系

- 任务总是归属于某个用户会话或系统会话
- 进程总是归属于某个 systemd unit 与 cgroup 层级
- execution token 必须同时绑定用户、会话、任务、时间窗口
- working memory 与执行进程强关联
- episodic / semantic / procedural memory 由 `sessiond` 或其下游存储组件治理
- shell 只负责展示与交互，不直接承担最终授权决定

## 9. 恢复模型

### 9.1 会话恢复

- shell 崩溃后，应允许重新挂接 `active` 或 `suspended` 任务
- 恢复不等于自动继续高风险动作
- 恢复时必须校验 execution token 是否仍有效

### 9.2 任务恢复

允许恢复的任务类型：

- 长时间推理
- 下载 / 同步
- 文档处理
- 低风险后台自动化

默认不自动恢复的任务类型：

- 破坏性文件操作
- 外部转账 / 提交 / 发送类操作
- 输入注入与自动点击类操作
- 系统更新与配置变更

### 9.3 自愈边界

AIOS 可以做：

- 诊断失败原因
- 选择替代工具路径
- 提供修复建议
- 在受限边界内执行 runbook

AIOS 不应做：

- 无审批地放大权限
- 在未知状态下重复高风险操作
- 因自愈逻辑而逃逸审计链

## 10. v1 必须冻结的接口语义

- `session.create`
- `session.resume`
- `session.close`
- `task.plan`
- `task.approve`
- `task.execute`
- `task.cancel`
- `task.audit.get`
- `runtime.infer`
- `runtime.route`
- `memory.read`
- `memory.write`
