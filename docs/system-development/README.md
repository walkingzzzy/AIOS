# AIOS 系统开发方案

**版本**: 1.2.0  
**更新日期**: 2026-03-08  
**状态**: Route B / 系统开发主线

---

本目录已根据 `AI操作系统设计与实现研究.md` 重新校准。  
其目标不是继续扩展 Electron 应用，而是把 AIOS 定义并推进为 **AI 原生操作系统 / 系统软件平台**。

这次修订后的主线不再只强调“系统服务 + 壳层”，而是进一步补入了 AI 操作系统真正缺失的六个系统维度：

1. **异构算力抽象**：CPU / GPU / NPU 不再被视为固定硬件，而是被视为可调度的运行时资源
2. **AI 运行时内核**：把 `LLM-as-Kernel` 落到工程上，解释为“认知控制平面”，而不是替代 Linux 内核
3. **多维记忆模型**：工作记忆、情景记忆、语义记忆、程序记忆必须进入系统设计
4. **多模态系统感知**：屏幕、音频、输入、摄像头等不只是外设，而是 AI 原生输入面
5. **协议化智能体通信**：MCP / A2A / 本地 IPC 必须共同构成 AIOS 的通信骨架
6. **可信混合计算**：本地优先，但必须为受证明、可审计、可禁用的可信云卸载预留架构

## 统一结论

- AIOS 的目标是 **系统开发**，不是应用开发
- AIOS v1 的现实形态是 **Linux-first、image-based、system-service-first** 的 AI-native 发行版 / 系统平台
- `LLM-as-Kernel` 在 AIOS 中是**认知控制平面的架构抽象**，不是替代 Linux ring0 内核
- 当前 `packages/client` 与 `packages/daemon` 仅是**过渡期兼容层原型**
- 所有未来设计都必须优先回答：它属于 `system`、`runtime`、`service`、`shell`、`device` 还是 `compat`

## 本轮修订吸收的关键研究结论

- AIOS 必须正面解决 CPU / GPU / NPU 异构调度，而不能只做用户态编排
- 上下文窗口、KV Cache、显存预算和情景记忆不是附加功能，而是 AI OS 的内生资源模型
- MCP / A2A 不能只当“插件协议”，而应纳入系统级 IPC 与工具调用治理框架
- 多模态感知必须下沉到系统设备与壳层，不应仅作为某个 app 的功能
- 可信云卸载必须遵守本地优先、可证明、可审计、可禁用、可回退
