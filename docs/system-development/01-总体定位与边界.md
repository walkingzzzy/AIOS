# 01 · 总体定位与边界

## 1. 项目定义

AIOS 被重新定义为：

> **一个 AI 原生操作系统 / 系统软件平台**，其目标是在系统层引入 AI 作为默认入口、任务编排器、能力调度器、运行时资源协调器和权限策略参与者。

它不再被定义为：

- 一个桌面 AI 助手应用
- 一个 Electron 客户端产品
- 一个“控制其他应用”的自动化壳
- 一个 MCP 工具聚合前端
- 一个纯协议层或 Agent 框架产品

## 2. AIOS 的核心问题

AIOS 要解决的不是“如何让 AI 更方便地点按钮”，而是：

1. 如何让 AI 成为系统一级入口
2. 如何把 CPU / GPU / NPU 抽象为可声明、可调度的 runtime 资源
3. 如何让工作记忆、长程记忆、知识记忆形成系统级模型
4. 如何让系统能力以 capability 形式被治理
5. 如何让任务、权限、审计、恢复形成系统闭环
6. 如何在原生能力与兼容层之间建立明确边界
7. 如何在本地优先前提下安全地扩展到可信云推理

## 3. 核心原则

### 3.1 system-first
所有设计首先服务于系统能力，而不是页面体验。

### 3.2 runtime-first
AIOS 不是只有 agent 编排，还必须拥有运行时资源抽象、模型路由与记忆管理。

### 3.3 local-first
关键控制路径默认在本地系统内完成，远端 AI 只能参与推理或受证明的受限卸载，不替代系统控制边界。

### 3.4 capability-first
AIOS 的最小治理单元不是页面、按钮或应用，而是 capability。

### 3.5 multimodal-native
屏幕、音频、图像、输入流不是附属数据，而是系统原生输入面。

### 3.6 recoverable-by-design
所有高风险操作必须支持失败检测、恢复、回滚或隔离。

### 3.7 compat-is-secondary
兼容层很重要，但它是桥接层，不是 AIOS 的定义中心。

## 4. 能力域定义

| 域 | 含义 |
|----|------|
| `system.*` | 操作系统基础能力 |
| `runtime.*` | 模型执行、队列调度、内存预算、推理路由、卸载控制 |
| `service.*` | 第一方系统服务能力 |
| `shell.*` | 会话、窗口、工作区、通知、启动器 |
| `device.*` | 硬件、传感器与多模态设备抽象 |
| `compat.*` | 浏览器、旧应用、办公软件、外部桥接 |
| `professional.*` | 专业域工作流与软件桥接 |

## 5. 非目标

以下事项不再作为 AIOS 主线目标：

- 继续以 Electron 主界面为产品中心
- 把“支持更多桌面应用控制”当成核心里程碑
- 用 GUI 自动化定义协议语义
- 把 Daemon + Client 打包发布视为“AIOS 正式版”
- 只做 Agent 框架式的工具聚合
- 只做聊天入口而忽略系统镜像、运行时和恢复链

## 6. 当前仓库的正确定位

当前仓库只能被解释为：

- **编排原型**：验证 AI 路由、任务流和会话语义
- **兼容层原型**：验证浏览器 / 文件 / 窗口等兼容能力
- **协议原型**：验证 MCP / A2A / JSON-RPC 的互联形态
- **系统开发前期控制台**：帮助团队在 Route B 初期完成调试和过渡

它不能被解释为：

- AIOS 系统镜像
- AIOS 原生壳层
- AIOS 运行时内核
- AIOS 完整权限与恢复系统
