## 1. 目标

本文不是架构愿景，也不是规格复述。  
它回答的是项目管理层面最直接的五个问题：

1. 当前到底做到哪一步了
2. 接下来按什么顺序推进
3. 每条主线当前是什么状态
4. 哪些任务已经完成、正在做、尚未开始、被阻塞
5. 每个阶段的退出条件和责任域是什么

## 2. 状态定义

| 状态 | 含义 |
|------|------|
| `Done` | 已有仓库产物或已通过阶段验收 |
| `In Progress` | 已开始，有明确产物，但未形成闭环 |
| `Planned` | 已进入正式计划，但尚未开工 |
| `Blocked` | 已识别依赖，当前无法推进 |
| `Deferred` | 不取消，但不占当前主线资源 |

## 3. 进度口径说明

本文中的完成度是**工程估算**，基于当前仓库证据，而不是 story point 或工时统计。  
评估口径如下：

- `0%`：只有想法，仓库无对应产物
- `10%`：有文档与目录骨架
- `25%`：有 schema / profile / unit / 配置骨架
- `50%`：有最小可运行实现
- `75%`：有测试、有 bring-up 验证、有恢复路径
- `100%`：进入稳定支持矩阵并长期维护

4. **provider runtime 覆盖仍不足**：已有 `system-files-provider`、shell control provider 与 compat runtime，但正式 device / bridge provider 仍不足
5. **runtime backend 仍缺更强 budget enforcement 与实机加速验证**：`runtimed` 已接 wrapper、GPU budget accounting 与 GPU -> CPU fallback，但 GPU/NPU 仍缺真实硬件级验证
6. **policy / audit 已形成 store + query + scoped approval + operator-facing evidence export 闭环**：已有 token / audit store / audit query、capability catalog、prompt guard、RPC 级审批测试矩阵，以及 high-risk token single-use consume / reuse 拒绝；新增 approval `target_hash` / `constraints` 绑定、`approval-scope-mismatch` 审计证据，以及统一 evidence export 的 approval scope 视图、mismatch 导出、`updated/recovery` 域覆盖与 `release-signoff` 域自动 stitching `system-delivery-validation` / governance / release gate / optional real-machine sign-off artifact，当前剩余工作主要转向更多真实 nominated machine sign-off 与现场取证扩面
7. **device / `ui_tree` 仍是当前主线未闭合项**：PipeWire / input / `ui_tree` 已形成 baseline、runtime helper、support matrix 与 shell 只读消费链，但真实 backend、完整 live tree 与正式 shell surface 仍未完成
8. **updated 已有 generic 平台 firmware slot bridge，但仍缺 vendor-specific firmware hook**：当前 generic x86_64 UEFI 路线已可用，仍需接入目标平台正式 firmware 工具
9. **integration tests 仍不足**：已有多条 smoke，但缺 registry/provider/image 自动化联调
10. **五团队计划已建立，但派生任务板与周同步仍需持续维护**：根计划、团队计划、任务清单、执行看板与 machine-readable YAML 已开始收口，但若后续只改其中一层，仍会重新引入口径漂移

## 8. 本周起应立即建立的管理机制

### 8.1 任务字段统一

每个正式任务至少带：

- `task_id`
- `track`
- `phase`
- `owner_role`
- `status`
- `depends_on`
- `entry_criteria`
- `exit_criteria`
- `repo_paths`
- `evidence`

当前 machine-readable 任务源：

- `aios/tasks/master-task-list.yaml`
- `aios/tasks/execution-board.yaml`

### 8.2 每周例会输出物

每周必须产出：

- 本周完成项
- 当前阻塞项
- 新增 ADR 需求
- 代码 / 文档证据链接
- 下周目标

固定模板：

- `docs/system-development/26-每周状态同步模板.md`

### 8.3 DoD（完成定义）

一个任务只有同时满足以下条件，才允许从 `In Progress` 进入 `Done`：

- 有仓库产物
- 有最小运行或验证证据
- 有回退 / 风险说明
- 有文档映射
