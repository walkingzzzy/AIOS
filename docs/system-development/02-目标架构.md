# 02 · 目标架构

## 1. 总体分层

```text
Hardware / Firmware
    ↓
Linux Kernel / Drivers / HMM / cgroup v2
    ↓
Boot / Image / Recovery / Update
    ↓
System Manager (systemd)
    ↓
AIOS Core Planes
  - Control Plane: aios-agentd / aios-sessiond / aios-policyd
  - Runtime Plane: aios-runtimed
  - Device Plane: aios-deviced
  - Trust Plane: aios-updated / audit / attestation
    ↓
AIOS Shell / Wayland Compositor
    ↓
Compat Providers / Code Sandboxes / Sandboxed Workloads
    ↓
Trusted Cloud Offload (Optional, Attested, Stateless)
    ↓
User Workloads
```

## 2. 架构总原则

### 2.1 Linux 是执行根，AI 是认知根

- Linux 负责资源控制、进程隔离、设备与启动链
- AIOS Cognitive Kernel 负责意图分解、运行时调度、记忆治理与能力决策

### 2.2 `LLM-as-Kernel` 是控制平面抽象

AIOS 应把研究文档中的 `LLM-as-Kernel` 落地为：

- `LLM syscall` → 推理与规划请求
- `Memory syscall` → 上下文 / 记忆读写
- `Storage syscall` → 长期数据与知识存取
- `Tool syscall` → 外部工具与 provider 调用
- `Device syscall` → 屏幕、音频、输入等采集或控制

这些并不是 Linux ring0 syscalls，而是 AIOS 在用户态系统服务层提供的**认知系统调用抽象**。

## 3. 核心系统服务

### 3.1 `aios-agentd`

负责：

- 意图分析与查询分解
- 任务规划与重规划
- agent topology 选择
- 路由策略选择：规则路由 / 语义路由 / LLM 路由 / 成本路由
- OODA 风格反馈闭环与失败回退

支持的任务模式：

- direct model call
- single agent with tools
- supervisor pattern
- plan-and-execute

`hierarchical teams` 仅作为 P3 以后预留，不进入 v1 主线。

### 3.2 `aios-runtimed`

负责：

- CPU / GPU / NPU 资源抽象
- 本地模型包装器与后端适配
- 队列调度、优先级与长短任务隔离
- 连续批处理、KV cache 预算、内存水位保护
- 本地 / 可信云推理路由决策
- 模型执行的速率限制、降级与熔断

推荐能力：

- MLFQ 式队列或等价分级机制
- request budget / memory budget / timeout budget
- CPU-only fallback
- 资源不足时拒绝执行，而不是系统级 OOM

### 3.3 `aios-sessiond`

负责：

- 用户会话
- 上下文持久化
- 任务与消息生命周期
- 工作流恢复
- 记忆系统治理

其内部至少区分四类记忆：

- working memory：当前任务 scratchpad、短时上下文、执行中间态
- episodic memory：会话 / 任务历史
- semantic memory：知识快照、嵌入索引、长期知识
- procedural memory：工具 schema、策略模板、工作流规范

### 3.4 `aios-policyd`

负责：

- capability policy
- 审批与确认
- 风险分级
- 审计日志
- prompt injection / taint source 处理
- 本地 / 云端执行边界判定

### 3.5 `aios-deviced`

负责：

- 输入输出设备抽象
- 显示、音频、蓝牙、网络、摄像头等设备能力
- UI tree 与屏幕像素并存的系统级屏幕感知
- 多模态事件聚合：音频、输入、视觉、通知、窗口

### 3.6 `aios-updated`

负责：

- 系统升级
- 健康检查
- 版本切换
- 回滚与恢复
- 自愈 runbook 与异常恢复入口

## 4. AI Shell

AI Shell 不是一个桌面应用，而是 AIOS 的系统壳层。  
其职责包括：

- AI launcher
- workspace 管理
- window / focus 管理
- notification 面板
- task surface / 审批界面
- session restore surface
- 多模态交互入口：文本、语音、屏幕上下文

## 5. 兼容层与代码沙箱

兼容层承担三类职责：

1. 与旧应用、浏览器、办公软件、SaaS 对接
2. 在 AIOS 原生能力尚未完成前，作为过渡性桥接
3. 为动态代码编排提供严格受限的执行场

兼容层包括：

- browser providers
- office providers
- MCP / A2A bridges
- GUI automation fallback
- code sandbox runtime

## 6. 数据与控制流

```text
User Intent / Multimodal Input
  ↓
AI Shell
  ↓
aios-agentd
  ↓
aios-policyd ← capability / risk / approval / taint
  ↓
aios-sessiond ← context / memory / history
  ↓
aios-runtimed ← local model / budget / queue / route
  ↓
system/service/shell/device/compat providers
  ↓
Result + Event + Audit + Recovery Hooks
```

## 7. 本地与可信云边界

AIOS 默认本地优先，但允许预留“可信云卸载”能力。其前提是：

- 远端节点经过证明或等价信任校验
- 远端执行是可禁用的
- 高敏感上下文默认不外发
- 卸载决策可审计
- 失败时自动回退到本地或安全降级路径

## 8. 技术路线建议

- 内核：Linux
- 服务管理：systemd
- 壳层：Wayland compositor / shell
- 核心语言：Rust / Go
- 原型与过渡层：TypeScript / Python
- 运行时引擎：vLLM 类高吞吐推理后端或等价能力
- 传输：Unix Socket / D-Bus / JSON-RPC / MCP / A2A
- 镜像：mkosi / image builder + atomic update strategy
