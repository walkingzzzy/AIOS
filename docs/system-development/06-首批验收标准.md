# 06 · 分阶段验收标准

## 1. 说明

本文件服务于**完整 AIOS 系统建设**。  
验收的目标不是压缩目标范围，而是保证每个阶段都能形成真实的系统闭环，并为后续阶段提供稳定前提。

## 2. Phase 0 验收

- 所有文档统一使用系统开发口径
- AIOS 的定义明确区分 Linux Kernel 与 Cognitive Kernel
- 新目录与重构方案存在
- 团队对 `legacy` / `compat` / `system` / `runtime` 边界达成一致
- `runtime.*` 与记忆模型完成冻结
- 基线能力与条件能力的表达方式完成冻结

## 3. Phase 1 验收

- 开发镜像可在 `QEMU` 启动
- 有基础日志输出
- 有基础恢复入口
- `runtimed` 能以 CPU-only 模式拉起最小推理包装器
- systemd units 能拉起核心 skeleton
- first-boot 初始化能够安全完成或重新进入修复路径

## 4. Phase 2 验收

- `agentd` 能接收并执行最小任务
- `runtimed` 能接收推理请求并执行本地路由决策
- `sessiond` 能记录会话、任务与最小记忆对象
- `policyd` 能做最小审批
- 核心服务通过本地 IPC 串联
- MCP discovery 或等价 provider discovery 跑通最小闭环
- execution token 与最小审计事件可联动验证

## 5. Phase 3 验收

- AI Shell 能启动
- 能展示 launcher、workspace、notification
- 能承载任务审批与任务状态可视化
- provider registry 可解析 system / shell / device / compat provider
- 至少有文件、导出目标、屏幕共享三类 portal 可工作
- working / episodic / semantic / procedural memory 至少有骨架实现

## 6. Phase 4 验收

- capability policy 可生效
- 高风险能力必须进入审批链
- prompt injection / taint source 有基本处理策略
- code sandbox 与 compat 权限隔离能生效
- 审计日志能关联 session / task / capability / route
- 可执行更新、失败可回滚、恢复入口可导出诊断包

## 7. Phase 5 验收

- 至少两类多模态主线闭环可在正式支持硬件上稳定工作
- 至少一条 GPU 加速路径在声明支持硬件上成立，且 CPU fallback 保持可用
- Tier 1 至少两台硬件通过基础图形、更新回滚与 policy / audit 验证
- queue / budget / timeout / memory budget 有稳定默认值与拒绝策略

## 8. Phase 6 验收

- `ui_tree` 或等价结构化图形来源至少在一个正式支持栈上稳定成立
- `local-npu` 若宣称支持，必须在对应 `hardware profile` 上可验证
- trusted cloud offload 若启用，attestation 失败时必须拒绝执行
- 条件能力均有明确支持矩阵、降级策略与审计语义

## 9. Phase 7 验收

- 开发者预览、产品预览、稳定版标准明确分层
- 支持矩阵、已知限制、升级与恢复策略文档冻结
- compat layer 不再混淆为系统本体
- 安全、审计、恢复与发布工程形成长期维护基线
