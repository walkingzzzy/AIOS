# 03 · 仓库重构方案

## 1. 重构目标

当前仓库结构仍然带有明显的“应用原型”痕迹。  
Route B 需要把仓库改造成**系统工程 + AI runtime 工程结构**。

## 2. 目标目录结构

```text
aios/
├── image/                 # 镜像、启动链、恢复、更新
├── services/              # 核心系统服务
│   ├── agentd/
│   ├── runtimed/
│   ├── sessiond/
│   ├── policyd/
│   ├── deviced/
│   └── updated/
├── runtime/               # 模型包装器、队列调度、KV cache、路由策略
├── shell/                 # AI Shell / compositor / UX surface
├── policy/                # capability policy、token、审计规范、taint 模型
├── compat/                # 浏览器、办公、MCP/A2A、旧应用桥接、代码沙箱
├── hardware/              # HAL、设备 profile、bring-up 记录
├── sdk/                   # SDK、schema、共享协议类型
├── observability/         # 日志、事件、metrics、trace、诊断包格式
├── docs/                  # 文档
└── legacy/                # 过渡期控制台与原型
    ├── client/
    ├── daemon/
    └── cli/
```

## 3. 现有模块迁移建议

| 现有位置 | 新位置 | 说明 |
|---------|--------|------|
| `packages/client` | `legacy/client` | 过渡期控制台 |
| `packages/daemon` | `legacy/daemon` → `services/agentd` / `services/runtimed` / `services/sessiond` | 先保留，再逐步抽取 |
| `packages/cli` | `legacy/cli` | 调试工具 |
| `packages/shared` | `sdk` | 协议类型和共享 schema |
| `adapters/*` | `compat/*` 或 `services/*` | 依能力域归位 |
| 推理后端封装 | `runtime/*` | 不应继续混杂在 daemon 内 |

## 4. 第一批拆分动作

### 4.1 从 daemon 抽离的内容

- task orchestration → `services/agentd`
- session persistence / history → `services/sessiond`
- approval / confirmation / audit → `services/policyd`
- model routing / queue / budget → `services/runtimed`
- screen / window / device provider glue → `services/deviced` 或 `compat/*`
- provider registry interface → `sdk` + `compat`

### 4.2 暂不抽离的内容

- Electron IPC glue
- preload 相关接口
- 纯 UI 控制逻辑
- 仅供 legacy demo 使用的过渡代码

## 5. 新增的工程边界

### 5.1 `runtime/` 边界

这里存放：

- model wrapper
- continuous batching
- memory budget / KV cache accounting
- local / remote route policy
- scheduling queue implementation

这里不应存放：

- shell UI
- 具体业务工作流
- 与 legacy UI 强耦合的状态逻辑

### 5.2 `hardware/` 边界

这里存放：

- bring-up checklist
- device capability profiles
- HAL / sensor 接入说明
- QEMU / Tier 1 机器配置
- 支持矩阵与条件能力声明

### 5.3 `observability/` 边界

这里存放：

- metrics schema
- trace event schema
- diagnostic bundle format
- crash / recovery evidence model

## 6. 仓库治理规则

从重构开始，所有新模块必须在 PR 中明确：

- 模块归属层级
- 它属于 control plane、runtime plane、device plane 还是 compat plane
- 是否属于系统主线
- 依赖边界
- 资源边界：CPU / GPU / NPU / memory / network
- 信任边界：local、sandbox、remote、attested remote
- 与现有原型代码的关系
- 未来是否可从 `legacy` 删除
- 它是基线能力还是条件能力
- 若为条件能力，其支持前提：hardware profile、shell stack、provider support、policy state

## 7. ADR 要求

从 Route B 开始，以下改动必须补 `ADR`：

- 新系统服务的引入或拆分
- 新 capability 域的引入
- 新的本地 / 云端推理路线
- 新的沙箱、代码执行、远端协议桥接方式
- 新的镜像、更新或恢复策略
- 新的 shell stack / compositor 路线
- 核心本地 IPC 与 bridge 边界调整
- 条件能力支持矩阵的正式引入或外部承诺
