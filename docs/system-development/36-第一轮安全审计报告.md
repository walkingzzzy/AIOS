# 36 · 第一轮安全审计报告

**版本**: 1.0.0  
**更新日期**: 2026-03-14  
**状态**: Draft Final

---

## 1. 审计目标

本轮审计面向 AIOS 当前仓库基线，目标是回答：

1. 当前高风险控制面是否已有最小安全闭环
2. 哪些安全能力已经能进入 release gate 口径
3. 哪些缺口仍然阻止产品预览或稳定版宣称

## 2. 审计范围

本轮覆盖：

- `policyd` approval / token / taint / audit
- `sessiond` portal handle binding
- `system-files-provider` 的 handle / token 消费边界
- `runtime-local-inference-provider` 的 token-verified façade
- cross-service observability / audit 基线

本轮不覆盖：

- Tier 1 实机安装/恢复现场操作安全
- vendor-specific firmware backend
- 真实 NPU / trusted offload 生产链路

## 3. 采用的仓库证据

- `cargo test -p aios-portal -p aios-sessiond -p aios-system-files-provider`

- `python3 scripts/test-provider-fs-smoke.py --bin-dir aios/target/debug`
- `python3 scripts/test-runtime-local-inference-provider-smoke.py --bin-dir aios/target/debug`
- `python3 scripts/test-cross-service-health-smoke.py --bin-dir aios/target/debug --delivery-manifest out/aios-system-delivery/manifest.json`

## 4. 已成立的安全基线

### 4.1 控制面授权边界

- 高风险 capability 已进入 approval gate
- `approval_ref` 与 token request context 已做绑定校验
- revoked / timed-out approval 不可继续签发 token

### 4.2 token 与签名

- execution token 已具备签名校验
- token key 路径已收紧为拒绝 symlink、0600/0700 权限与原子写入
- `target_hash` 已进入 token model 与 verify path

### 4.3 taint / prompt injection

- prompt injection baseline 已能拒绝高风险 destructive flow
- 上游 `taint_summary` 现已能传播到 `policy.evaluate`、approval 与 execution token
- tainted 内容不会在 `policyd` 中被静默清洗掉

### 4.4 审计链

- `policyd` 已有 segmented audit store、rotation、retention 与跨段 query
- approval create / resolve、token issue / verify、policy evaluate 都会进入 audit
- observability sink 已能镜像关键 audit record

### 4.5 portal / provider 消费边界

- `portal.handle.lookup` / `revoke` 已按 `session_id` / `user_id` 绑定
- `system-files-provider` 查 handle 时会携带 token 上下文
- `file_handle` / `directory_handle` 已验证真实 target 并拒绝 symlink

## 5. 本轮发现

### 5.1 高风险 token 仍缺单次消费语义

严重度：`High`

规范已要求高风险 token 必须单次消费，但当前实现主要覆盖签名、TTL、context 与 target 绑定，尚未看到消费后作废的仓库证据。

### 5.2 approval scope 细粒度绑定已补齐基线

严重度：`Medium`

跟进到 2026-03-14，`approval` 已补齐 `target_hash` + `constraints` 作用域绑定：`policy.evaluate` / `approval.create` 现会把 scoped target 与约束一并写入 approval record，`policy.token.issue` 会对 approval scope 与 token request 做结构化比对，并在 `approval_target_mismatch` / `approval_constraints_mismatch` 场景写入 `approval-scope-mismatch` 审计证据。`agentd` 的 delete flow、`system-files-provider`、screen capture chooser/provider 与相关 smoke 也已切到这条精细 scope 链路。

### 5.3 审计导出面仍未统一

严重度：`Medium`

`policyd` 已有 query/store。跟进到 2026-03-14，仓库已新增 `scripts/build-audit-evidence-report.py` 与 `scripts/test-audit-evidence-export-smoke.py`，现可把 `sessiond` / `policyd` / `runtimed` 的 control-plane 证据，以及 shell / provider / compat / device retained artifact、`updated` apply / rollback / recovery retained evidence、默认 `Tier 1 hardware evidence` baseline，与 `release-signoff` 域下的 `system-delivery-validation` / governance / release gate / optional real-machine `hardware-validation-evidence.json` 一并统一导出为 operator-facing report；最新导出也已补入 approval scope、target-bound/constraint-bound 统计与 scope mismatch 证据，并开始把 compat 的 shared audit sink、centralized-policy 命中数、token-verified 记录与 per-provider timeout/deny 汇总纳入同一报告。当前剩余缺口已从“完全没有统一导出”进一步收敛为“真实 nominated machine sign-off 与更高保真现场取证尚未并入同一 export / gate 闭环”。

### 5.4 新增高风险路径的审计补齐机制仍依赖人工推进

严重度：`Medium`

当前已有 `X-OBS-001` 持续任务，但尚未看到“新增高风险 capability 自动阻塞 release gate，直到 audit event 就位”的强制机制。

跟进状态（2026-03-14）：

- 已新增 `tests/observability/high-risk-audit-coverage.yaml`
- 已新增 `scripts/test-high-risk-audit-coverage-smoke.py`
- 已纳入 `tests/observability/validation-matrix.yaml`、`.github/workflows/aios-ci.yml` 与 `docs/RELEASE_CHECKLIST.md`

这意味着新增 `risk_tier=high` capability，或团队 A 的 `update.apply` / `update.rollback` / `recovery.bundle.export` 若未补齐审计覆盖矩阵，将直接在 `validate` / `release gate` 阶段失败。  
因此，本项风险已从“纯人工推进”收敛为“仓库级 blocking gate + control-plane operator-facing evidence export baseline 已落地”，但更广覆盖面的统一 audit evidence export 仍未完全闭环。

## 6. 结论

本轮审计结论为：

- AIOS 已具备 `Developer Preview` 级别的安全控制面基线
- AIOS 尚未达到 `Product Preview` 或 `Stable Release` 所需的安全完备度

进入下一阶段前，建议优先关闭：

1. 将统一 audit / evidence export 从当前 control-plane + shell / provider / compat / device / updated / hardware / release-signoff 覆盖继续扩展到真实 nominated machine sign-off 与更高保真现场取证
2. 持续维护新高风险路径的 audit gate，并把 operator-facing evidence export 一并收口
