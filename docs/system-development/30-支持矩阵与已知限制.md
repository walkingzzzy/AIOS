# 30 · 支持矩阵与已知限制

**版本**: 1.2.0  
**更新日期**: 2026-03-14  
**状态**: `P5-HW-005` + `P7-REL-006` frozen baseline

## 1. 目标

本文冻结 AIOS 当前团队 A 交付链的支持边界、已知限制与对外宣称口径。

它回答四个问题：

1. 当前哪些平台只完成了仓库/QEMU 验证
2. 哪些平台已经具备 platform media / bring-up kit 交付能力
3. 哪些能力可以被宣称为“当前支持”，哪些只能宣称为“条件能力”或“待实机验证”
4. 发布说明、runbook 与 operator 文档应如何引用这些结论

## 2. 支持状态定义

| 状态标签 | 含义 | 当前可否对外交付 |
|----------|------|------------------|
| `Repo/QEMU Validated` | 仓库级、QEMU 级或纯 smoke 级验证已完成 | 仅可宣称内部开发/验证基线 |
| `Platform Media Exportable` | 可导出 installer media、recovery media、system image 与 bring-up kit | 可交付给 operator 做 bring-up，但不能宣称稳定硬件支持 |
| `Hardware Validation Required` | 平台声明要求至少一份真实机器安装、首启、update/rollback/recovery 证据 | 未补证据前不得宣称“硬件支持已成立” |
| `Hardware Validated` | 至少两次不同 `boot_id`、安装与回退证据齐全，并通过最终硬件验证报告 | 才能进入正式支持清单 |
| `Conditional Capability` | 能力依赖具体 hardware profile、runtime profile、device backend 或策略前提 | 必须在发布说明中单独声明，不得视为全平台默认能力 |

## 3. 当前平台支持矩阵

| 平台 | Tier | 架构 | `updated` profile | runtime profile | 平台介质导出 | 当前验证口径 | 当前对外口径 |
|------|------|------|-------------------|-----------------|--------------|--------------|--------------|
| `qemu-x86_64` | Tier 0 | `x86_64` | `/usr/share/aios/updated/platforms/qemu-x86_64/profile.yaml` | `default-local` + `default-route` | 不作为独立 handoff 目标 | 已通过仓库/QEMU bring-up、recovery、installer-media、cross-reboot、system-delivery-validation | 仅可宣称内部开发基线，不可等同于 Tier 1 硬件支持 |
| `generic-x86_64-uefi` | Tier 1 参考目标 | `x86_64` | `/usr/share/aios/updated/platforms/generic-x86_64-uefi/profile.yaml` | `/etc/aios/runtime/default-runtime-profile.yaml` + `default-route` | 支持 `build-aios-platform-media.py` 导出 | 已有平台 profile、firmware bridge、platform media、bring-up kit 与 smoke；仍缺真实机器证据 | 可宣称“平台介质与 bring-up kit 已冻结”，不可宣称“稳定硬件支持已成立” |
| `nvidia-jetson-orin-agx` | Tier 1 条件目标 | `aarch64` | `/usr/share/aios/updated/platforms/nvidia-jetson-orin-agx/profile.yaml` | `/usr/share/aios/runtime/platforms/nvidia-jetson-orin-agx/default-runtime-profile.yaml` + `default-route` | 支持 `build-aios-platform-media.py` 导出 | 已有平台 profile、firmware bridge、managed worker 资产与 bring-up kit；仍缺真实机器证据 | 仅可宣称 bring-up kit 与 repo 级平台路径，不可宣称 release-grade Jetson 支持 |

- `framework-laptop-13-amd-7040` -> `generic-x86_64-uefi`

## 4. 能力域支持边界

| 能力域 | 当前冻结状态 | 当前可宣称口径 | 备注 |
|--------|--------------|----------------|------|
| Boot / first-boot / recovery image | `Repo/QEMU Validated`；Tier 1 为 `Platform Media Exportable` | 可宣称仓库级和平台交付基线已建立 | Tier 1 仍缺真实安装、首启和恢复证据 |
| Installer media / recovery media 写盘脚本 | `Platform Media Exportable` | 可宣称 operator handoff 资产已存在 | 不等同于硬件安装成功记录 |
| `update.check` / `update.apply` / `update.rollback` / `update.health.get` | 契约、状态回写与 smoke 已冻结 | 可宣称 update/recovery 契约已成立 | 硬件级 firmware hook、timeout、失败注入仍未闭环 |
| `recovery.surface.get` + recovery UI / CLI | JSON/RPC、`client.py`、`panel.py`、`shellctl` 路径已成立 | 可宣称 operator / shell prototype 可消费恢复面 | 这不是最终正式 GUI shell |
| `recovery.bundle.export` 与诊断包解释 | exporter、schema、解读文档已存在 | 可宣称可导出并解释 diagnostic bundle | live exporter 尚未写入 sample 中全部扩展字段 |
| CPU-only runtime fallback | 已有默认 runtime profile 与 fallback 语义 | 可宣称这是当前全平台默认运行时下限 | 仍需更多实机性能与稳定性证据 |
| `local-gpu` / `local-npu` | `Conditional Capability` | 只能按 profile 单独声明 | `local-cpu` 仍是全平台默认；`local-gpu` 的当前首选平台判断已冻结在 `38` 与 `aios/runtime/gpu-backend-support-matrix.yaml`，其中 Jetson runtime profile 已打包 worker 资产，但未形成实机 release-grade 证据 |
| Wi-Fi / 蓝牙 / 音频 / 摄像头 | `Conditional Capability` | 只能宣称“profile 声明/待验证” | 当前没有稳定的 release-grade device backend 结论 |
| `ui_tree` | `In Progress` + `Conditional Capability` | 只能宣称评估与只读链路已建立 | 完整 live tree 与 fully-qualified 支持矩阵未闭环 |
| Tier 1 硬件正式支持名单 | `Not Yet Established` | 不得宣称已有稳定硬件清单 | 正式机器清单已冻结，但 `P5-HW-003/004` 的实机证据仍待补齐 |

## 5. 当前已知限制

- 截至 2026-03-14，仓库内没有任何一个 Tier 1 平台达到 `Hardware Validated` 状态。
- `generic-x86_64-uefi` 是仓库级参考 bring-up 目标，不是已冻结的厂商量产机型。
- `framework-laptop-13-amd-7040` 已被冻结为正式 x86_64 Tier 1 机器，但目前仍只具备 nomination 状态，没有实机签收证据。
- `nvidia-jetson-orin-agx` 已有 platform profile、firmware bridge 与 managed worker 资产，但仍只有 bring-up kit 级别结论。
- `qemu-x86_64` 的成功记录只能证明仓库/QEMU 闭环，不能替代实机安装、回退和恢复签收。
- `updated` 的 vendor-specific firmware hook 仍缺真实目标机器上的失败注入、超时与回退验证。
- `recovery-surface` 已有 CLI、panel skeleton 与 live smoke，但正式 GUI/compositor 集成仍未到 release-grade。
- `diagnostic bundle` 的 schema、sample 与 operator 文档已经冻结，但 live exporter 还没有写入 sample 中全部扩展字段。
- Linux/x86_64 镜像构建在 `docker buildx` 可用时已优先走 container-native delivery；当 docker 不可用但已存在 `out/aios-delivery-container-target/debug` 时，也会先复用缓存的 container-native 二进制目录。上述偏好已由 `scripts/test-image-build-strategy-smoke.py` 输出到 `out/validation/image-build-strategy-report.{json,md}`；但完整镜像产线仍保留 `host-bin-dir` 最终回退路径，尚未完全收敛到纯容器内原生可重复构建。
- device / multimedia 相关能力仍以 native probe、state bridge、stub 或条件声明为主，暂不能纳入“默认稳定支持”。

## 6. 发布与交付规则

1. 发布说明必须同时引用本文和 [32-版本兼容矩阵.md](32-版本兼容矩阵.md)。
2. 若某平台没有真实硬件验证报告，只能写成 `Repo/QEMU Validated` 或 `Platform Media Exportable`，不得写成“支持该硬件”。
3. release gate 现默认强制校验 `out/validation/tier1-hardware-evidence-index.json`；若要用真实实机签收替换或补充默认基线，必须附带 `hardware-validation-evidence.json`，并通过 `--hardware-evidence-index` 指向对应证据。
4. 任何 Optional / Experimental 能力都必须和具体 `hardware profile`、runtime profile、device backend 或策略前提一起声明。
5. 新增或修改平台 profile 时，必须同步更新本文、[27-升级与恢复交付Runbook.md](27-升级与恢复交付Runbook.md) 与 [32-版本兼容矩阵.md](32-版本兼容矩阵.md)。

## 7. 当前结论

当前最准确的对外交付表述应为：

> AIOS 已形成 `qemu-x86_64` 的仓库/QEMU 验证基线，并已冻结 `generic-x86_64-uefi` 与 `nvidia-jetson-orin-agx` 的 platform media / bring-up kit 交付口径；但截至 2026-03-14，Tier 1 实机安装、更新失败回退与恢复取证证据仍未闭环，因此尚不能宣称稳定硬件支持名单已经成立。
