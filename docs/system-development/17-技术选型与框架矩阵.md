# 17 · 技术选型与框架矩阵

**版本**: 1.0.0  
**更新日期**: 2026-03-08  
**状态**: 实现层基线文档 / Route B

---

## 1. 目标

前序文档主要回答“AIOS 是什么、边界在哪里、能力如何分层”。  
本文开始回答工程执行中最常见、也最容易失控的四个问题：

1. 每个系统域准备用什么技术开发
2. 使用什么框架与运行时
3. 哪些已经冻结，哪些仍需 ADR
4. 每项技术当前处于什么状态

本文不是 marketing 式“技术名词清单”，而是**面向仓库实施与人力安排的技术选型矩阵**。

> 进度口径说明：本文里的“当前仓库状态 / 实装度”用于记录技术选型冻结时的成熟度基线，不作为 2026-03-14 的权威进展统计。当前实现进度请以 `docs/IMPLEMENTATION_PROGRESS.md`、`18-开发主计划与任务状态.md`、`19-实现映射与当前进度.md`、`22-执行任务看板.md` 为准。

## 2. 状态定义

| 状态 | 含义 | 对实现的约束 |
|------|------|--------------|
| `Accepted` | 已在文档 / ADR 层冻结 | 新实现默认按此路线推进 |
| `Selected` | 已确定为首选实现，但 ADR 尚未补完 | 可进入设计和原型，不宜大规模扩散 |
| `Candidate` | 推荐方向，仍需实验或 ADR | 只能做 spike / PoC |
| `Experimental` | 预研项或条件能力 | 不得作为基线前提 |
| `Blocked` | 已识别依赖缺失或风险过高 | 需要先解决阻塞再推进 |

## 3. 总体技术策略

### 3.1 语言策略

- **系统主线**：Rust
- **legacy / 管理脚本 / 辅助工具**：TypeScript、Python、Shell
- **原则**：
  - 新的 core service 默认不用 Node.js 做长期运行时
  - Python 仅用于模型实验、离线处理、开发脚本、性能对比
  - TypeScript 保留在 `legacy/`、开发工具和桥接控制台

### 3.2 架构策略

- **内核与执行根**：Linux
- **系统管理器**：systemd
- **镜像交付**：image-based
- **核心控制面通信**：本地 IPC 优先
- **外部互操作**：MCP / A2A / API bridge 进入 registry 与 audit
- **完整系统表达**：基线能力 + 条件能力声明

|------|------------------|------|------|--------------|------|
| 基础 OS | Linux LTS | 无 | `Accepted` | 仅文档冻结 | 执行根 |

| 大对象与缓存 | 文件系统对象存储 + `model-cache` | 对象数据库 | `Selected` | 仅布局文档 | 简化回滚与恢复 |
| 代码沙箱 | `bubblewrap` 进程沙箱 | 容器 / Firecracker | `Candidate` | 尚无实现 | Linux 桌面友好 |
| 轻量计算沙箱 | WASI / `wasmtime` | 纯脚本解释器 | `Experimental` | 尚无实现 | 适合短生命周期工具 |

## 5. 分域详细说明

### 5.1 镜像、启动、更新

**推荐路线**

- 启动：UEFI + `systemd-boot`
- 镜像构建：`mkosi`
- 分区与首次启动：`systemd-repart`、`systemd-firstboot`、`systemd-growfs`
- 更新：`systemd-sysupdate`
- 恢复：专用 recovery deployment + 诊断包导出

**理由**

- 与 `Linux-first + image-based + systemd-first` 的主线完全一致
- 官方文档对 golden image、first-boot、identity reset、A/B update 都有明确建议
- 比自建安装器 / 自建升级器更容易形成长期维护基线

**当前状态**

- 文档已冻结
- `aios/image/` 已有 `mkosi.conf`、`repart.d/`、`sysupdate.d/`、overlay 与脚本
- 尚缺 boot 测试、first-boot 配置、可启动镜像验证

**必须补齐的文件**

- `aios/image/mkosi.conf` 或等价目录
- `aios/image/sysupdate.d/*.conf`
- `aios/image/repart.d/*.conf`
- `aios/image/boot/` 相关配置
- `scripts/boot-qemu.sh` 或等价脚本

### 5.2 核心服务开发框架

**推荐路线**

- 语言：Rust
- async runtime：Tokio
- schema / serialization：Serde + JSON Schema
- 日志：Tracing
- 错误：thiserror / anyhow
- 配置：TOML / YAML + schema 校验

**原因**

- Rust 更适合长期运行的系统 daemon、强类型 schema、资源约束与边界安全
- Tokio 适合多路 IPC、超时控制、异步任务与后台 worker
- JSON Schema 已在仓库中出现，适合 descriptor / handle / profile / token 统一化

**落地约束**

- 所有新 core service 默认使用 Rust
- Go 仅在必须引入的外部生态场景再评估
- Node.js 不再作为长期系统守护进程主实现

### 5.3 核心 IPC 与系统总线

**推荐路线**

分两层：

1. **核心控制面 RPC**：Unix Domain Socket + JSON-RPC 2.0
2. **桌面集成 / 系统对象集成**：D-Bus

**设计理由**

- JSON-RPC over UDS 便于本地抓包、调试、版本演进与跨语言兼容
- D-Bus 适合与 desktop notifications、portal、session bus、system bus 对接
- 这符合“核心本地控制面优先，bridge 另算”的新 ADR

**不推荐**

- 不以 MCP / A2A 作为核心服务之间唯一通信机制
- 不让 GUI fallback 反向定义系统主语义

### 5.4 Shell / Compositor / UI

**建议分两层建设**

1. **Compositor 层**：Smithay（Rust）
2. **Shell 控件层**：GTK4 / libadwaita，先承担审批面板、恢复面板、设置与调试视图

**原因**

- Smithay 提供 Rust Wayland compositor 构建能力，适合长期系统壳层方向
- 但完整 shell 的交互面板、设置页、恢复页若全部自绘，早期工程成本过高
- 用 GTK4 / libadwaita 承担壳层控制面 UI，可提高开发速度并保持 Linux 桌面一致性

**当前状态**

- 文档已提出 Wayland shell 方向
- 仓库无 compositor 源码、无 shell 客户端源码、无 UI 原型

**必须补 ADR**

- `Shell Stack ADR`
- 是否引入 XWayland
- shell panel 与 compositor 是否同进程

### 5.5 Runtime 与推理后端

**原则**

- `runtimed` 不是某个单一推理框架的别名，而是**统一调度与预算服务**
- 后端必须插件化：`local-cpu`、`local-gpu`、`local-npu`、`attested-remote`

**建议后端路线**

- `local-cpu`：轻量本地后端（`llama.cpp` 类路线）
- `local-gpu`：vLLM
- `local-npu`：按硬件 profile 接入 vendor-specific backend
- `attested-remote`：后期条件能力

**理由**

- `local-cpu` 是 bring-up 与恢复模式的底线
- vLLM 在 Linux GPU/CPU 场景有成熟 serving 能力，但硬件支持仍取决于具体平台
- NPU 与可信远端都不适合先写成统一默认能力

### 5.6 存储、记忆与状态

**推荐拆分**

- 结构化元数据：SQLite（`sessiond`）
- 大对象 / 原始模态摘要 / 模型缓存：文件系统目录
- 审计导出：JSON Lines / tar.zst 诊断包
- 向量 / 长期语义索引：P5 以后再决定是否引入专门组件

**原因**

- 单机系统优先，不需要一开始就引入网络数据库
- SQLite + 文件系统最适合恢复、快照、离线导出与本地调试
- 真正的语义索引组件应等 usage pattern 明确后再选型

### 5.7 沙箱与 compat

**推荐分级**

- 代码沙箱：`bubblewrap` 为主
- 极轻量工具执行：WASI / `wasmtime` 为辅
- 第三方 GUI 工作负载：类 Flatpak 权限模型
- 浏览器 / SaaS / Office：独立 compat provider

**原因**

- 代码执行和应用兼容不是一回事，不能共用一套最宽权限模型
- `bubblewrap` 符合 Linux 桌面常见最小权限模式
- WASI 适合短脚本、纯计算、无宿主副作用的工具任务

### 5.8 可观测性与测试

**推荐路线**

- 日志：journald + `tracing`
- 审计：结构化 JSON schema
- trace：统一 trace event schema
- bring-up：QEMU snapshot + boot 验证脚本
- 服务单测：Rust test / nextest
- 集成测试：场景驱动的 QEMU / IPC / policy 测试

**当前状态**

- 审计与 trace schema 只有占位
- 无测试脚本、无 CI 验证链、无 QEMU 自动化

## 6. 未冻结但必须尽快冻结的决策

以下是当前最关键、最影响后续代码的“未冻结技术决策”：

1. `mkosi + systemd-sysupdate` 是否正式成为 image/update 路线
2. Shell 是否正式采用 Smithay
3. Shell 控件层是否采用 GTK4 / libadwaita
4. 核心 RPC 是否正式采用 JSON-RPC over UDS
5. `sessiond` 是否正式采用 SQLite + 文件对象存储
6. `local-cpu` baseline 后端选择哪条路线
7. 代码沙箱是否以 `bubblewrap` 为主

这些都应该在进入大量编码前补 ADR。

## 7. 当前进度摘要（技术角度）

| 领域 | 冻结度 | 实装度 | 说明 |
