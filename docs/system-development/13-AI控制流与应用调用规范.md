# 13 · AI 控制流与应用调用规范

**版本**: 1.0.0  
**更新日期**: 2026-03-08  
**状态**: P0/P1 核心规格

---

## 1. 目标

本文件专门回答 AIOS 中最关键的三个实现问题：

1. AI 如何进行系统操作
2. AI 如何调用应用端与兼容层
3. AI 应如何被配置

其核心目的，是把“AI 是入口”落地成**受控、可审计、可恢复的执行链**，而不是停留在“聊天 + 自动化”的产品层描述。

## 2. 总体原则

### 2.1 AI 不直接执行系统操作

在 AIOS 中：

- AI 负责理解意图、生成计划、选择能力、解释结果
- Linux Kernel 负责资源控制、进程隔离、设备与启动链
- AIOS Core Services 负责授权、调度、执行、审计、恢复

因此，AI **不是 root**，也**不是信任根**。

### 2.2 AI 不直接调用应用 UI

应用端调用必须遵循分层优先级：

1. 原生 provider / 原生 API
2. portal / broker / IPC
3. 协议桥接（MCP / A2A / provider bridge）
4. GUI automation fallback

只有在更高层没有结构化能力时，才允许降级到更低层。

### 2.3 AI 配置不是 prompt 配置

AI 配置必须是系统级分层配置，至少覆盖：

- 模型配置
- 路由配置
- 记忆配置
- 权限配置
- 交互配置
- 应用 / provider 配置

### 2.4 完整系统的“完整”不等于全部路径无条件成立

在完整 AIOS 中，控制闭环必须完整；  
但具体执行路径是否成立，仍取决于：

- 当前 `hardware profile`
- 当前图形栈 / shell stack
- 当前 provider 是否声明支持
- 当前策略是否允许

因此：

- `ui_tree`、`local-npu`、trusted offload 属于正式能力域
- 它们必须被显式声明、可审计、可降级
- 它们不能被假定为“每个任务、每台机器、每个应用默认都可用”

## 3. AI 如何进行系统操作

## 3.1 基本执行链

AIOS 中所有系统操作必须走统一控制流：

```text
User Intent / Multimodal Input
  ↓
AI Shell
  ↓
aios-agentd
  ↓
Task Plan + Capability Resolution
  ↓
aios-policyd
  ↓
Execution Token
  ↓
aios-sessiond / aios-runtimed / aios-deviced / provider
  ↓
Result + Audit + Recovery Hooks
```

任何绕过这条链的“捷径执行”都视为架构错误。

## 3.2 系统操作的九步模型

### 第 1 步：意图接收

输入来源可以是：

- 文本输入
- 语音输入
- 屏幕上下文
- 当前窗口 / 工作区状态
- 设备事件

这些输入先进入 `AI Shell`，再交给 `aios-agentd`。

### 第 2 步：任务规划

`aios-agentd` 将自然语言或多模态输入转为：

- `intent`
- `task_id`
- `plan_version`
- `candidate_capabilities`
- `target_objects`
- `expected_side_effects`

如果任务复杂，可选择：

- `direct`
- `tool-calling`
- `supervisor`
- `plan-execute`

### 第 3 步：能力解析

AI 不直接说“我去删文件”，而是把任务解析为 capability，例如：

- `system.file.read`
- `system.file.write`
- `device.control.display.brightness.set`
- `shell.window.focus`
- `compat.browser.navigate`
- `runtime.infer.local`

### 第 4 步：策略评估

`aios-policyd` 对每个 capability 做统一判断：

- 谁在发起
- 属于哪个会话
- 操作对象是什么
- 是否 tainted
- 风险级别是什么
- 在本地、沙箱还是受证明远端执行
- 是否需要审批

### 第 5 步：审批与 token 签发

若能力属于 `R3` / `R4` 或触发高风险条件，则进入审批链。  
审批通过后，由 `policyd` 签发一次性或短时 `execution token`。

执行端只认 token，不认 UI 状态，不认“模型说已经批准了”。

### 第 6 步：执行位置决策

能力被批准后，还需决定由谁执行：

- `sessiond`：上下文、记忆、任务状态变更
- `runtimed`：模型推理、路由、预算控制
- `deviced`：设备控制、采集、多模态事件
- `shell` 组件：窗口、焦点、通知、工作区
- `compat provider`：旧应用、浏览器、办公软件桥接
- `code sandbox`：受限动态代码执行

### 第 7 步：执行与观察

执行过程必须具备：

- 进度事件
- 失败事件
- 结果结构化返回
- 可取消
- 可超时
- 可限额

### 第 8 步：审计落盘

每次 capability 执行都要留下：

- 谁发起
- 对什么对象执行
- 在哪里执行
- 是否审批
- 执行结果
- 是否回退
- 是否触发恢复

### 第 9 步：恢复或重规划

失败后，AIOS 可以：

- 给出失败原因
- 请求人工确认
- 尝试低风险替代路径
- 进入恢复 runbook
- 放弃执行并保留证据

AIOS 不可以：

- 因失败而放大权限
- 无限制重试高风险动作
- 脱离审计链偷偷修复

## 3.3 三类系统操作路径

### A. 原生系统路径

适用于：

- 文件、网络、进程、系统状态
- 屏幕亮度、音量、通知、电源
- 工作区、窗口、焦点管理

路径：

`AI Shell -> agentd -> policyd -> system/service/device/shell provider`

### B. 兼容应用路径

适用于：

- 浏览器
- Office
- 旧桌面软件
- 外部 SaaS / 协议桥接

路径：

`AI Shell -> agentd -> policyd -> compat provider`

### C. 代码沙箱路径

适用于：

- 数据清洗
- 临时解析逻辑
- 小型转换与推理辅助脚本

路径：

`AI Shell -> agentd -> policyd -> code sandbox`

该路径默认高于普通工具调用风险，因为会引入动态执行。

## 3.4 示例：系统操作如何执行

### 示例 1：调节亮度

用户说：`把屏幕亮度调到 60%`

控制流：

1. `agentd` 识别为 `device.control.display.brightness.set`
2. `policyd` 判断为低风险设备控制
3. 签发 token
4. `deviced` 调用底层显示接口
5. 返回结果并写审计

### 示例 2：批量删除文件

用户说：`清空下载目录里的临时文件`

控制流：

1. `agentd` 生成候选文件列表与删除计划
2. `policyd` 识别为高风险文件写操作
3. 进入审批链
4. 审批后签发 token
5. 文件 provider 执行删除
6. 审计记录写入，必要时生成恢复引用

### 示例 3：恢复失败任务

用户说：`继续刚才失败的导出`

控制流：

1. `sessiond` 查找最近失败任务
2. `agentd` 生成恢复计划
3. `policyd` 判断是否允许恢复
4. 若为高风险操作，则再次审批
5. 执行恢复或给出新计划

## 4. 如何调用应用端

## 4.1 应用端不是系统本体

在 AIOS 中，应用端的正确定位是：

- 第一方扩展能力
- 兼容层桥接能力
- 被托管工作负载

它们不定义系统主语义，只提供被治理的能力面。

## 4.2 应用调用优先级

### Level 1：原生能力优先

如果 AIOS 已经有原生 provider，则优先使用：

- `system.*`
- `service.*`
- `shell.*`
- `device.*`
- `runtime.*`

例：

- 打开通知中心 → `shell.notification.open`
- 调节音量 → `device.control.audio.volume.set`

### Level 2：结构化应用接口

如果原生能力没有覆盖，则优先应用的结构化接口：

- portal / broker
- D-Bus / local IPC
- app-specific provider
- 官方 API

例：

- 导出文档为 PDF
- 查询邮件列表
- 打开浏览器新标签页

### Level 3：协议桥接

如果应用不在本机、或需要跨系统对接，则使用：

- MCP
- A2A
- SaaS bridge
- remote provider

要求：

- schema 必须可验证
- 对端必须有来源信息
- 调用必须纳入 policy 与 audit
- 它们是互操作 / 兼容路径，不是 `agentd`、`sessiond`、`policyd`、`runtimed` 之间核心本地 IPC 的替代

### Level 4：GUI 自动化兜底

只有前 3 级都不可用时，才允许使用：

- UI tree 自动化
- 屏幕视觉识别
- 鼠标键盘输入注入

此类调用默认高风险，并需要：

- 独立授权
- 可视化提示
- 超时与取消机制
- 自动化失败回退
- 在支持矩阵中明确标注适用的 shell / app / 图形栈范围

## 4.3 应用调用的四类对象

### 4.3.1 生命周期对象

- `compat.lifecycle.launch`
- `compat.lifecycle.focus`
- `compat.lifecycle.minimize`
- `compat.lifecycle.close`

### 4.3.2 数据对象

- `compat.document.open`
- `compat.document.export`
- `compat.data.query`
- `compat.data.write`

### 4.3.3 功能对象

- `compat.mail.send`
- `compat.browser.navigate`
- `compat.office.export_pdf`
- `compat.calendar.create_event`

### 4.3.4 自动化对象

- `compat.ui.inspect`
- `compat.ui.click`
- `compat.ui.type`
- `compat.ui.capture`

在 Route B 中，应用端对象统一收敛到 `compat.*` 或更具体的 provider 域，不再扩展新的 `app.*` 命名空间。

## 4.4 应用调用的标准控制流

```text
Intent
  ↓
agentd
  ↓
resolve provider capability
  ↓
policyd
  ↓
compat provider / provider bridge / portal
  ↓
provider result
  ↓
audit + session update
```

## 4.5 compat provider 的最小要求

每个 compat provider 至少要声明：

- 支持的 capability 列表
- 文件权限需求
- 网络权限需求
- 屏幕 / 输入 / 剪贴板权限需求
- 是否允许远端对接
- 超时与重试策略
- 可观测事件

## 4.6 典型调用策略

### 浏览器

优先级：

1. `compat.browser.*` 原生桥接
2. 浏览器扩展或协议接口
3. MCP / Web bridge
4. GUI automation fallback

### Office

优先级：

1. `compat.office.*` 原生 provider
2. 文件格式桥接 / export API
3. 自动化桥接
4. GUI fallback

### 本地 GUI 软件

优先级：

1. 生命周期 + portal
2. app-specific provider
3. 协议桥接
4. UI 自动化

### 远端 SaaS

优先级：

1. 官方 API / OAuth provider
2. MCP bridge
3. A2A / workflow bridge
4. 浏览器自动化

## 5. 如何配置 AI

## 5.1 AI 配置的六层模型

### Layer 1：模型配置

控制：

- 默认模型
- 本地模型池
- 远端模型池
- 最大上下文
- 温度 / 推理模式
- 超时 / 重试 / 预算

### Layer 2：路由配置

控制：

- direct / tool-calling / supervisor / plan-execute 的选择规则
- 规则路由 / 语义路由 / LLM 路由 / 成本路由优先级
- 本地优先还是混合路由
- CPU-only fallback 是否开启

### Layer 3：记忆配置

控制：

- working memory 大小与生命周期
- episodic memory 的保留期限
- semantic memory 的索引策略
- procedural memory 的版本管理
- 高敏感数据是否允许进入 long-term memory

### Layer 4：权限配置

控制：

- 默认允许的 capability
- 必须审批的 capability
- 永久禁止的 capability
- 本地 / 沙箱 / 远端执行边界
- tainted 输入的处理策略

### Layer 5：交互配置

控制：

- 默认交互入口
- 是否启用语音
- 是否启用屏幕上下文
- 是否启用多模态捕获
- 是否显示审批面板、通知、恢复入口

### Layer 6：应用与 provider 配置

控制：

- 已启用 provider 列表
- 每个 provider 的权限声明
- MCP / A2A 对端列表
- compat provider 白名单 / 黑名单
- automation fallback 是否允许启用

## 5.2 配置优先级

AIOS 推荐采用以下优先级：

```text
System Defaults
  < Organization Policy
  < Hardware Profile
  < User Profile
  < Workspace Profile
  < Session Overrides
  < Task-local Overrides
```

规则：

- 下层可以细化行为
- 下层不能突破上层安全约束
- 权限配置优先级高于模型配置
- 组织策略优先级高于用户偏好

## 5.3 配置对象模型

AI 配置不应收敛为单一大文件，而应拆成多个对象：

- `runtime-profile`
- `route-profile`
- `memory-profile`
- `policy-profile`
- `shell-profile`
- `provider-profile`

## 5.4 推荐配置示例

```yaml
runtime_profile:
  default_backend: local
  local_models:
    - qwen-local-14b
    - deepseek-local-7b
  remote_models:
    - gpt-4.1
  cpu_fallback: true
  timeout_ms: 30000
  memory_budget_mb: 6144

route_profile:
  default_topology: tool-calling
  allow_supervisor: true
  semantic_router: true
  llm_router: true
  cost_router: true
  prefer_local: true

memory_profile:
  working_memory_window: 32000
  episodic_retention_days: 30
  semantic_index_enabled: true
  procedural_memory_versioned: true
  allow_sensitive_long_term_memory: false

policy_profile:
  require_approval:
    - compat.ui.input.inject
    - system.file.bulk_delete
    - device.capture.microphone.record
  deny:
    - system.security.disable_policy
  remote_offload_default: disabled
  taint_mode: strict

shell_profile:
  voice_input: true
  screen_context: true
  show_approval_panel: true
  show_recovery_actions: true

provider_profile:
  enabled:
    - compat.browser
    - compat.office
    - mcp.bridge
  automation_fallback: manual-only
```

## 5.5 AI 配置的生效方式

### 启动时加载

用于：

- system default
- organization policy
- hardware profile

### 登录时加载

用于：

- user profile
- shell profile
- provider preference

### 会话时加载

用于：

- workspace profile
- session route preference
- temporary memory rules

### 任务时覆盖

用于：

- 单任务模型选择
- 单任务超时
- 单任务 provider 约束
- 单任务审批强化

## 5.6 哪些配置不能交给用户随意修改

以下配置必须受 `policyd` 或系统管理员约束：

- 高风险能力是否自动批准
- 远端可信执行是否默认开启
- taint 检查是否关闭
- 审计是否关闭
- code sandbox 的资源上限是否取消
- compat provider 是否越权访问文件或设备

## 6. 三个问题的标准答案

## 6.1 AI 如何进行系统操作

标准答案：

> AI 不直接操作系统，而是通过 `agentd -> policyd -> sessiond/runtimed/deviced/provider` 的受控链路申请并执行 capability。

## 6.2 如何调用应用端

标准答案：

> AI 不直接操作应用 UI，而是优先调用原生 provider、portal、结构化接口与协议桥接；只有在这些能力都不存在时，才允许降级到 GUI automation fallback。

## 6.3 如何配置 AI

标准答案：

> AI 配置是分层系统配置，不是单一 prompt 配置；它必须同时覆盖模型、路由、记忆、权限、交互与 provider 六类对象，并服从系统与策略优先级。
