# 09 · 镜像、安装、更新与恢复规范

**版本**: 1.1.0  
**更新日期**: 2026-03-08  
**状态**: P0/P1 必备规格

---

## 1. 目标

AIOS 必须是一个**可启动、可安装、可升级、可回滚、可恢复**的系统，而不是一组用户态程序集合。

## 2. 推荐技术路线

v1 推荐固定为以下路线：

- 基线内核：Linux
- 启动链：UEFI + `systemd-boot` 或等价现代启动链
- 镜像构建：`mkosi` 或等价 image builder
- 服务管理：`systemd`
- 分区与首启处理：`systemd-repart`、`systemd-firstboot`、`systemd-growfs`
- 更新策略：image-based atomic update
- 回滚策略：保留上一个可启动 deployment
- 本地推理资产：与系统 deployment 分离管理

## 3. 镜像类型

### 3.1 开发镜像

- 格式：`raw` / `qcow2`
- 目标：QEMU / UTM / VMware / Parallels
- 用途：bring-up、服务联调、恢复链验证

### 3.2 安装镜像

- 格式：安装 ISO 或预制裸盘镜像
- 目标：x86_64 UEFI 笔记本 / 台式机
- 用途：开发者预览与硬件验证

### 3.3 更新工件

- 格式：完整系统 deployment、分层镜像或等价快照工件
- 要求：具备签名、版本号、兼容性元数据、回滚信息

### 3.4 运行时资产工件

包括但不限于：

- model wrapper
- runtime policy bundle
- provider package
- knowledge / embedding index snapshot

要求：

- 与系统 deployment 分离版本管理
- 高敏感资产具备来源记录与审计
- 更新失败不得破坏系统可启动性

## 4. 启动链

推荐启动顺序：

`UEFI -> bootloader -> kernel -> initrd -> rootfs -> systemd -> AIOS core services -> AI Shell`

要求：

- `agentd` / `runtimed` / `sessiond` / `policyd` 必须由 system manager 拉起
- shell 不得早于核心策略与运行时服务处于不可用状态时承担完整交互责任
- 恢复模式必须在正常 shell 之前可达

## 5. 分区与数据布局

v1 推荐的最小逻辑布局：

- `ESP`：启动分区
- `system`：只读或准只读系统部署区
- `var`：日志、状态、缓存、数据库
- `home`：用户数据
- `model-cache`：模型缓存、KV / embedding 相关大对象缓存
- `recovery`：恢复工具与最小诊断环境（可逻辑合并）

如采用双 deployment 或 A/B 变体，则必须额外满足：

- 至少保留一个上一个可启动版本
- 回滚不依赖网络
- 回滚后审计链不中断

## 6. 首次启动

首次启动必须完成以下动作：

- 生成唯一 `machine-id`
- 生成或注册设备标识
- 初始化本地账户或开发者默认配置
- 初始化 `var`、`model-cache` 与必要 state 目录
- 写入镜像版本元数据
- 标记 first-boot 完成点
- 生成本地 / 可信云执行策略初始值

要求：

- 镜像中不得预置共享随机种子或共享机器标识
- 首次启动失败时，下次启动仍应能进入 first-boot 修复路径
- 若可信云能力默认关闭，必须保持关闭状态直至显式启用

## 7. 更新模型

### 7.1 更新原则

- 更新必须是**原子切换**而非散装覆盖
- 更新失败必须可回滚
- 更新后必须有健康检查
- 更新前后必须保留审计记录
- 运行时资产更新不得破坏基本系统启动

### 7.2 更新对象

至少区分四类对象：

1. 系统 deployment
2. AIOS 核心服务版本
3. compat / provider 包
4. model / runtime 资产

其中：

- 系统 deployment 更新属于高风险操作
- compat/provider 更新不得影响系统可启动性
- model 文件或模型缓存更新不得与系统 deployment 强耦合
- 远端卸载配置变更必须可追踪、可回退

## 8. 健康检查

更新后最小健康检查项：

- 能进入启动链并到达登录或 shell
- `agentd`、`runtimed`、`sessiond`、`policyd` 正常启动
- 基础存储、日志、网络可用
- 审批链可工作
- 恢复入口仍可访问
- runtime budget / queue 至少可进入安全默认值

若未通过，则自动回滚到上一个可启动 deployment。

## 9. 恢复模式

AIOS v1 必须至少提供以下恢复能力：

- 回滚到上一个 deployment
- 导出日志与诊断包
- 禁用损坏的 compat/provider
- 禁用远端卸载并回到本地安全模式
- 重置损坏的 shell 用户态状态
- 进入最小维护终端或恢复 UI

恢复模式下默认禁止：

- 启动高风险自动化任务
- 自动外发敏感数据
- 直接信任上次 AI 计划并继续执行
- 自动启用可信云卸载

## 10. 自愈 runbook 原则

AIOS 可以内置 runbook 来帮助恢复：

- service restart
- cache cleanup
- compat disable
- runtime degrade
- route fallback

但必须遵守：

- 有审计
- 有权限边界
- 有失败退出条件
- 不可绕过 recovery / approval 语义

## 11. 开发阶段验收

### P1 验收

- 能构建 `QEMU` 开发镜像
- 能完成首启初始化
- 能拉起核心 services skeleton
- 能进入恢复入口

### P2 验收

- 可做一次原子更新
- 更新失败可回滚
- 回滚后系统与审计数据一致可读
- runtime 资产更新失败不影响系统进入安全模式

### P3 验收

- 有开发者安装介质
- 有版本兼容矩阵
- 有更新健康检查报告
- 有恢复模式与诊断包导出能力
