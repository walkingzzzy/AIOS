# 10 · 能力、策略与审计规范

**版本**: 1.1.0  
**更新日期**: 2026-03-08  
**状态**: P0 必备规格

---

## 1. 目标

本文件定义 AIOS 的治理核心：AI 不直接“拥有权限”，AI 只能申请 capability，由策略系统决定是否允许执行。

## 2. capability 定义

capability 是 AIOS 的最小治理单位。一个 capability 必须回答：

- 它作用于什么对象
- 它会产生什么影响
- 它的风险级别是什么
- 它需要什么前置条件
- 它是否需要审批与审计
- 它是否会触发本地 / 云端执行边界变化
- 它是否涉及多模态采集、代码执行或外部系统副作用

## 3. 命名规则

AIOS 统一使用以下主命名空间：

- `system.*`
- `runtime.*`
- `service.*`
- `shell.*`
- `device.*`
- `compat.*`
- `professional.*`

历史命名规则：

- `app.*` 仅保留为历史别名
- 新能力不得再进入 `app.*`
- 所有兼容层能力一律归入 `compat.*`

## 4. AIOS 认知系统调用抽象

为吸收研究文档中的 `LLM syscall / Memory syscall / Storage syscall / Tool syscall`，AIOS 内部统一抽象以下 capability 基元：

- `runtime.infer.*`
- `memory.read.*`
- `memory.write.*`
- `storage.read.*`
- `storage.write.*`
- `tool.invoke.*`
- `device.capture.*`
- `device.control.*`
- `runtime.route.*`

说明：

- 它们是 AIOS 的**认知系统调用抽象**
- 不是 Linux ring0 syscall
- 最终仍由用户态系统服务和策略系统治理

## 5. capability 元数据

每个 capability 至少包含以下字段：

- `id`
- `version`
- `domain`
- `summary`
- `risk_level`
- `target_schema`
- `preconditions`
- `approval_mode`
- `audit_tags`
- `recovery_class`
- `execution_location`：`local` / `sandbox` / `attested_remote`
- `data_classification`
- `latency_budget`
- `power_budget`
- `memory_budget`

## 6. 风险分级

### R0 · 无副作用读取

- 只读状态查询
- 不涉及敏感数据聚合

### R1 · 低风险用户态动作

- 单文件读写
- 本地界面导航
- 非敏感通知
- 低风险本地推理

### R2 · 中风险持久化动作

- 多文件修改
- 任务创建与计划保存
- 兼容层内的受限自动化
- 低敏数据的 embedding / 索引更新

### R3 · 高风险外部或破坏性动作

- 大量删除或覆盖
- 对外发送、发布、支付、提交
- 屏幕录制、麦克风采集、输入注入
- 兼容层跨边界调用
- 生成代码并执行

### R4 · 管理员级系统动作

- 系统更新与版本切换
- 用户与凭据管理
- 安全策略变更
- 恢复模式与系统级设备控制
- 改变本地 / 远端执行信任边界

## 7. 策略评估顺序

`subject -> session -> task -> capability -> target -> environment -> taint -> route -> risk -> approval -> execution token`

解释：

1. 验证主体是谁
2. 验证会话是否合法
3. 验证任务计划版本
4. 解析 capability 元数据
5. 校验目标对象范围
6. 校验设备、网络、锁屏、恢复模式等环境约束
7. 识别 tainted 输入来源
8. 判定本地、沙箱或受证明远端执行位置
9. 根据风险级别决定审批方式
10. 由 `policyd` 签发短时 execution token

## 8. token 模型

execution token 至少绑定：

- `user_id`
- `session_id`
- `task_id`
- `capability_id`
- `target_hash`
- `expiry`
- `approval_ref`
- `constraints`
- `execution_location`
- `taint_summary`

规则：

- token 只可短时使用
- token 不可被 UI 伪造
- token 不得跨用户、跨任务复用
- 高风险 token 必须单次消费
- execution location 变化后必须重新签发 token

## 9. 审批模型

### 9.1 审批触发条件

以下情况必须进入审批链：

- `R3` 与 `R4` 能力
- 目标对象超出原计划范围
- 外部网络副作用
- 凭据、设备、输入注入相关动作
- 恢复模式与系统配置变更
- 启用受证明远端执行
- 动态代码生成与执行

### 9.2 审批展示内容

审批 UI 不能只显示自然语言摘要，至少显示：

- capability 名称
- 目标对象
- 风险级别
- 是否可恢复
- 数据是否外发
- 是否涉及系统状态改变
- 执行位置：本地 / 沙箱 / 受证明远端
- 来源是否 tainted

## 10. 审计模型

每次 capability 执行必须产生审计事件，至少包含：

- `audit_id`
- `timestamp`
- `user_id`
- `session_id`
- `task_id`
- `capability_id`
- `target`
- `decision`
- `approval_ref`
- `executor`
- `result`
- `error_code`
- `recovery_ref`
- `route_state`
- `execution_location`
- `model_backend`
- `taint_summary`

审计要求：

- 审计事件默认不可静默关闭
- 高风险审计不可被普通用户删除
- 任务视图、会话视图、恢复视图都能引用同一审计链

## 11. 与 compat 的关系

compat provider 无权自定义系统主策略。它只能：

- 声明需要的 capability
- 在沙箱范围内请求授权
- 接收 `policyd` 下发的受限 token
- 返回结构化结果与来源信息

compat 不得：

- 自建平行审批体系
- 绕过审计
- 以“自动化脚本”名义获得系统级豁免
- 自行决定切换到远端可信执行域

## 12. v1 强制约束

- 所有高风险操作必须由 `policyd` 最终裁决
- shell 不是授权根
- model runtime 不是授权根
- legacy 工具不是授权根
- provider registry 不是授权根
- tainted 内容不得直接驱动高风险动作

## 13. 结论

AIOS 的权限模型必须从“谁能调用某个 API”升级到“谁、在什么会话、基于什么计划、对什么对象、以什么风险级别、在什么环境下、以什么执行位置、是否经审批后调用 capability”。
