## 1. 目标

本文把 `02-目标架构` 的“服务职责说明”扩展成真正可实施的详细设计。  
重点回答：

1. 每个核心服务用什么技术实现
2. 每个服务负责哪些模块、暴露哪些接口
3. 服务之间如何通信
4. 服务按什么顺序启动
5. 当前每个服务处于什么进度

## 2. 设计范围

本文覆盖以下核心服务：

- `agentd`
- `runtimed`
- `sessiond`
- `policyd`
- `deviced`
- `updated`

并补充两个跨服务子系统：

- provider registry
- audit / event pipeline

## 3. 统一技术基线

### 3.1 语言与运行时

- 语言：Rust
- async runtime：Tokio
- serialization：Serde JSON
- schema validation：JSON Schema
- logging / tracing：Tracing
- system integration：D-Bus + systemd unit

### 3.2 本地 IPC 约定

建议采用两层：

1. **核心服务 RPC**：Unix Domain Socket + JSON-RPC 2.0
2. **桌面对象与 portal 集成**：D-Bus

### 3.3 数据布局约定

建议约定：

- `/var/lib/aios/sessiond/`：会话、任务、记忆元数据
- `/var/lib/aios/policyd/`：审批状态、策略快照、token key material
- `/var/lib/aios/registry/`：provider registry 状态缓存
- `/var/cache/aios/model-cache/`：模型与 runtime 缓存
- `/var/log/aios/`：辅助导出日志（主日志仍进 journald）
- `/var/lib/aios/audit/`：结构化审计事件存储

## 4. 启动顺序与依赖图

### 4.1 推荐顺序

```text
systemd
  ↓
updated
  ↓
sessiond
  ↓
policyd
  ↓
runtimed
  ↓
agentd
  ↓
deviced
  ↓
AI Shell / provider / compat / portal
```

### 4.2 原则

- `updated` 负责确保系统当前 deployment 可用，尽早进入健康状态
- `sessiond` 与 `policyd` 要早于大多数交互面和高风险执行路径
- `runtimed` 可在 shell 前启动，但 shell 不应依赖其全部高级能力才进入基础交互
- `deviced` 可以晚于控制面主服务，但要早于多模态主路径

## 5. 服务详设

### 5.1 `agentd`

#### 角色

- 意图理解
- 任务规划
- capability 候选解析
- topology 选择
- 失败重规划

#### 推荐技术

- Rust + Tokio
- JSON-RPC over UDS
- 内部 planner / resolver / route-client 模块

#### 目录建议

```text
services/agentd/
├── src/
│   ├── main.rs
│   ├── config.rs
│   ├── rpc.rs
│   ├── planner.rs
│   ├── resolver.rs
│   ├── topology.rs
│   ├── execution.rs
│   └── errors.rs
├── service.yaml
└── units/
```

#### 输入

- `intent.submit`
- `task.replan`
- `task.resume`
- multimodal context refs
- provider candidates

#### 输出

- `task_plan`
- `candidate_capabilities`
- `route_request`
- `recovery_plan`

#### 关键接口（建议）

- `agent.intent.submit`
- `agent.task.plan`
- `agent.task.replan`
- `agent.task.resume`
- `agent.task.explain`

#### 强约束

- 不直接执行高风险 capability
- 不持久保存最终授权状态
- 不绕过 `policyd`
- 不直接持有长期主记忆写权限

#### 当前状态

- 仓库层级：`Partial Impl`
- 已有源码、最小 IPC skeleton、planner / resolver / RPC 骨架，缺上游联调

### 5.2 `runtimed`

#### 角色

- backend 抽象
- queue / budget / timeout / backpressure
- local / remote route policy
- inference worker 管理
- CPU fallback

#### 推荐技术

- Rust 主控服务
- backend plugin 机制
- 本地 worker 用子进程管理
- metrics / trace 事件输出

#### 目录建议

```text
services/runtimed/
├── src/
│   ├── main.rs
│   ├── config.rs
│   ├── rpc.rs
│   ├── scheduler.rs
│   ├── queue.rs
│   ├── budget.rs
│   ├── backend/
│   │   ├── mod.rs
│   │   ├── cpu.rs
│   │   ├── gpu.rs
│   │   ├── npu.rs
│   │   └── remote.rs
│   └── errors.rs
```

#### 输入

- `runtime.infer`
- `runtime.route`
- selected `runtime-profile`
- selected `route-profile`
- policy constraints

#### 输出

- inference result
- route decision
- budget event
- degrade / reject event

#### 关键接口

- `runtime.infer.submit`
- `runtime.route.resolve`
- `runtime.backend.list`
- `runtime.health.get`
- `runtime.queue.get`
- `runtime.budget.get`

#### 关键实现点

- `local-cpu` 必须长期存在
- `local-gpu` / `local-npu` 只能按 profile 激活
- 任何执行位置变化都要回写 route / audit state
- OOM 不能直接冒泡为系统级崩溃

#### 当前状态

- 仓库层级：`Partial Impl`
- 有 profile / schema，且已有 scheduler / queue / budget / backend skeleton，缺真实 worker

### 5.4 `policyd`

#### 角色

- capability policy evaluation
- 风险分级
- 审批状态机
- execution token 签发与验证
- taint source / prompt injection 防御
- audit event 生成

#### 推荐技术

- Rust + policy evaluator
- token signer/verifier
- SQLite 或本地密钥文件 + `0600` 权限管理

#### 目录建议

```text
services/policyd/
├── src/
│   ├── main.rs
│   ├── config.rs
│   ├── rpc.rs
│   ├── evaluator.rs
│   ├── token.rs
│   ├── approval.rs
│   ├── taint.rs
│   ├── audit.rs
│   └── errors.rs
├── rules/
├── keys/
└── units/
```

#### 关键接口

- `policy.evaluate`
- `policy.approval.request`
- `policy.approval.resolve`
- `policy.token.issue`
- `policy.token.verify`
- `policy.audit.append`

#### 强约束

- shell 不是授权根
- runtime 不是授权根
- bridge 不是授权根
- token 必须绑定 user / session / task / target / expiry / execution location

#### 当前状态

- 仓库层级：`Partial Impl`
- 已有 policy profile、token schema、evaluator / token issue / audit writer skeleton，缺 verify 与审批状态机

## 6. Provider Registry 的实现位置

v1 不建议立即新增独立的 `registryd`。  
建议做法：

- 先实现一个 `provider-registry` 共享库
- 由 `agentd` 调用 capability resolution
- 由 `policyd` 调用 provider trust / execution location filter
- provider 状态缓存与持久化可落到 `sessiond` 或独立本地 state 文件

原因：

- 当前服务数量已经足够多
- registry 更像横切子系统，而不是必须马上分裂成第 7 个常驻核心服务
- 等 provider 数量、健康检查复杂度、动态加载场景上来后，再决定是否独立 daemon

## 7. Audit / Event Pipeline

建议统一三类事件：

- `audit event`
- `trace event`
- `health / recovery event`

最小要求：

- 所有高风险 capability 都必须写 audit
- 所有降级、回退、拒绝都要写 trace / audit
- update / recovery / trusted offload 事件必须能跨服务串联

## 8. 当前实现阶段结论

当前所有核心服务都还处在**设计已明确、代码未开始**阶段。  
因此后续真实开发的第一步不应该是继续补“功能页”，而应该是：

1. 建 Rust workspace
2. 先实现 `sessiond` / `policyd` / `runtimed` 的最小 RPC
3. 跑通 QEMU image 与 systemd unit
4. 再把 shell / portal / provider 接上
