# 21 · 完整任务清单

**版本**: 1.1.4  
**更新日期**: 2026-03-14  
**状态**: 执行层主任务清单 / Route B

---

## 1. 目标

本文是 AIOS 的**完整任务清单**。  
它不再停留在 Phase、Backlog 或原则层，而是把 AIOS 的开发任务拆成可以逐项推进、逐项验收、逐项改状态的工作项。

适用范围：

- 系统主线开发
- image / boot / recovery
- core services
- shell / provider / portal
- policy / audit / sandbox
- multimodal / hardware / conditional capabilities
- release / ops / documentation maintenance

## 2. 状态定义

| 状态 | 含义 |
|------|------|
| `Done` | 已有明确仓库产物或文档产物，且阶段目标已经成立 |
| `In Progress` | 已开工，有明确产物，但未形成闭环 |
| `Planned` | 已进入主计划，但尚未开工 |
| `Blocked` | 已识别阻塞，依赖未满足 |
| `Deferred` | 不取消，但不占当前主线资源 |

## 3. Owner Role 定义

| Owner Role | 责任范围 |
|------------|----------|
| `PM/Architects` | 路线、ADR、阶段 gate、状态同步 |
| `Platform` | image、boot、update、recovery、QEMU、CI |
| `Systems` | core services、IPC、shared crates |
| `Runtime` | backend、queue、budget、route、fallback |
| `Security` | policy、token、approval、taint、audit、sandbox |
| `Shell` | compositor、launcher、approval UI、recovery UI |
| `Compat` | browser / office / bridge / provider runtime |
| `Device` | capture、multimodal、visible indicators |
| `QA/Validation` | bring-up、hardware matrix、regression、release validation |
| `Docs/Ops` | runbook、support matrix、release notes、diagnostics docs |

## 4. 使用规则

- 每个任务都必须有唯一 `Task ID`
- 每个任务必须能映射到仓库路径、文档或验证证据
- 任何任务从 `Planned` 进入 `Done` 前，至少要有：
  - 仓库产物
  - 最小验证证据
  - 文档同步
- 任何条件能力任务都必须显式标注支持前提

## 5. 关键路径

AIOS 的主关键路径如下：

1. 治理与 ADR 冻结
2. Image / Boot / QEMU bring-up
3. Rust workspace + core local IPC
4. `sessiond` / `policyd` / `runtimed` 最小闭环
5. `agentd` + provider registry + portal
6. shell / approval / recovery surface
7. update / rollback / diagnostics
8. multimodal / GPU / Tier 1 / conditional capabilities
9. productization / release / long-term maintenance
