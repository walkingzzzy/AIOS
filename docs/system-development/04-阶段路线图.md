# 04 · 阶段路线图

## 1. 节奏说明

本路线图服务于 **完整 AIOS 系统**，不是 MVP 压缩计划。  
因此本文件只给出**依赖顺序**与**参考工期**，而不是以 marketing 式固定周数承诺全部能力同步完成。

原则：

- 先闭合 `boot -> service -> shell -> policy -> execution -> audit -> recovery` 主环
- 再扩展 GPU / NPU、多模态、复杂图形栈、可信云等条件能力
- 终局蓝图不缩水，但阶段门槛必须尊重 bring-up 顺序与实现耦合

## 2. 0-24+ 月总览（参考工期）

| 阶段 | 时间 | 目标 |
|------|------|------|
| Phase 0 | 0-1 月 | 语义冻结、ADR 冻结、边界与仓库结构冻结 |
| Phase 1 | 1-3 月 | 可启动开发镜像 + `QEMU` Tier 0 + first-boot / logging / recovery 入口 |
| Phase 2 | 3-6 月 | `agentd` / `runtimed` / `sessiond` / `policyd` skeleton + 本地 IPC + 最小审计闭环 |
| Phase 3 | 6-9 月 | AI Shell alpha + provider registry + portal + 第一方 provider 最小执行闭环 |
| Phase 4 | 9-12 月 | 更新 / 回滚 / 恢复 + compat 隔离 + code sandbox + policy hardening |
| Phase 5 | 12-18 月 | 多模态输入面 + GPU 路线 + Tier 1 硬件收敛 + runtime / memory budget 稳定化 |
| Phase 6 | 18-24 月 | 条件能力闭环：`ui_tree` 支持栈、`local-npu`、trusted offload、复杂图形与设备增强 |
| Phase 7 | 24+ 月 | 产品预览 / 稳定版硬化：支持矩阵、发布工程、安全与运维基线长期稳定 |

## 3. Phase 0

- 冻结错误的“应用开发”叙事
- 吸收 `AI操作系统设计与实现研究.md` 的关键结论
- 明确 Linux Kernel 与 Cognitive Kernel 的双层关系
- 建立 `runtime` / `hardware` / `observability` 新边界
- 明确 `legacy` / `compat` / `system` / `runtime` 边界
- 冻结“基线能力”与“条件能力”的表达方式

## 4. Phase 1

- 建立镜像构建链
- 跑通 VM / QEMU 启动
- 建立日志与恢复入口
- 验证最小 rootfs
- 跑通 CPU-only 推理包装器 skeleton
- 拉起 `runtimed` 的最小 unit
- 跑通 first-boot、machine identity 与基础分区 / 状态目录初始化

## 5. Phase 2

- 建立 `agentd` skeleton
- 建立 `runtimed` skeleton
- 建立 `sessiond` skeleton
- 建立 `policyd` skeleton
- 用本地 IPC 串联最小工作流
- 建立 MCP / A2A / provider discovery 的最小骨架
- 建立 execution token、policy decision 与最小审计事件模型

## 6. Phase 3

- AI launcher
- workspace / focus / notification
- task surface
- 最小人工审批界面
- provider registry
- portal（至少文件、导出目标、屏幕共享三类）
- 第一方 system / shell / device provider 的最小闭环
- working / episodic / semantic / procedural memory 的最小实现

## 7. Phase 4

- capability token
- 审批与风险模型
- prompt injection / taint source 模型
- compat 权限隔离
- code sandbox alpha
- 更新包格式
- 健康检查
- 回滚机制
- 恢复模式与诊断包导出

## 8. Phase 5

- 屏幕 / 音频 / 输入多模态采集主线闭环
- GPU 加速路径与 CPU fallback 协同
- Tier 1 硬件收敛与支持矩阵建立
- runtime queue / budget / KV cache / memory budget 稳定化
- compat 与 shell / provider / audit 闭环收束

## 9. Phase 6

- `ui_tree` 在至少一个正式支持的图形栈 / 应用栈上稳定成立
- `local-npu` 在声明支持的硬件 profile 上成立
- trusted cloud offload 在 attestation、可禁用、可审计前提下进入 beta
- 复杂图形与外设增强能力进入 declared-support 模式

## 10. Phase 7

- 开发者预览与产品预览标准分离
- 稳定支持矩阵冻结
- 发布工程、更新工程与恢复工程长期化
- 安全审计、运维 runbook、自愈策略进入长期维护基线

## 11. 阶段顺序背后的原因

这份路线图吸收研究文档后有一个关键变化：

- AIOS 不能先做壳层，再补运行时
- AIOS 必须先有 **bootable image + runtime substrate + policy boundary**
- registry / portal 必须早于大规模 compat 扩展
- 多模态、复杂 `ui_tree`、NPU、可信云都必须建立在上述基线上
- “完整系统”要求保留全部目标能力，但不要求它们在同一时间点、同一硬件、同一图形栈无条件成立
