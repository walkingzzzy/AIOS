# 04 · 阶段路线图

## 1. 0-24 周总览

| 阶段 | 时间 | 目标 |
|------|------|------|
| Phase 0 | 0-2 周 | 语义冻结、研究吸收、架构与边界冻结 |
| Phase 1 | 2-6 周 | 可启动开发镜像 + QEMU 基线 + CPU-only runtime skeleton |
| Phase 2 | 6-10 周 | 核心系统服务与 runtime substrate skeleton |
| Phase 3 | 10-15 周 | AI Shell alpha + 多模态输入骨架 + 会话记忆体系 |
| Phase 4 | 15-20 周 | capability policy / audit / sandbox / trusted offload beta |
| Phase 5 | 20-24 周 | 更新、恢复、自愈 runbook 与开发者预览 |

## 2. Phase 0

- 冻结错误的“应用开发”叙事
- 吸收 `AI操作系统设计与实现研究.md` 的关键结论
- 明确 Linux Kernel 与 Cognitive Kernel 的双层关系
- 建立 `runtime` / `hardware` / `observability` 新边界
- 明确 `legacy` / `compat` / `system` / `runtime` 边界

## 3. Phase 1

- 建立镜像构建链
- 跑通 VM / QEMU 启动
- 建立日志与恢复入口
- 验证最小 rootfs
- 跑通 CPU-only 推理包装器 skeleton
- 拉起 `runtimed` 的最小 unit

## 4. Phase 2

- 建立 `agentd` skeleton
- 建立 `runtimed` skeleton
- 建立 `sessiond` skeleton
- 建立 `policyd` skeleton
- 用本地 IPC 串联最小工作流
- 建立 MCP / A2A / provider discovery 的最小骨架

## 5. Phase 3

- AI launcher
- workspace / focus / notification
- task surface
- 最小人工审批界面
- 屏幕 / 音频 / 输入多模态采集骨架
- working / episodic / semantic / procedural memory 的最小实现

## 6. Phase 4

- capability token
- 审批与风险模型
- prompt injection / taint source 模型
- compat 权限隔离
- code sandbox alpha
- trusted cloud offload beta（可禁用）

## 7. Phase 5

- 更新包格式
- 健康检查
- 回滚机制
- 自愈 runbook / OODA 反馈流程
- 开发者预览版本标准

## 8. 阶段顺序背后的原因

这份路线图吸收研究文档后有一个关键变化：

- AIOS 不能先做壳层，再补运行时
- AIOS 必须先有 **bootable image + runtime substrate + policy boundary**
- 多模态、MCP / A2A、可信云都必须建立在上述基线上
