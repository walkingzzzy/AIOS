## 一、项目概述

### 1.1 核心定位

**AIOS Protocol 是让 AI 通过自然语言控制操作系统和软件的开放标准协议。**

```
用户 (自然语言) → AI 引擎 (意图理解) → AIOS Protocol (执行控制) → 操作系统/软件
```

### 1.2 开放设计理念

AIOS Protocol 采用**开放设计**，不限制技术方案：

| 维度 | 开放性 |
|------|--------|
| **实现语言** | Python、TypeScript、Go、Rust、Java 等 |
| **传输方式** | stdio、HTTP、WebSocket、Unix Socket、gRPC |
| **部署模式** | 本地、云端、Serverless、容器化 |
| **生态集成** | MCP 桥接、A2A 桥接、OpenAPI 转换 |

### 1.3 与现有方案的差异

| 方案 | 定位 | 控制方式 | 局限性 |
|------|------|---------|--------|
| **Claude Computer Use** | 桌面自动化 | 截图 + 坐标点击 | 慢、易出错、依赖视觉 |
| **OpenAI Operator** | 浏览器自动化 | GUI 交互 | 仅限浏览器、已停用 |
| **MCP** | 工具调用协议 | API 调用 | 需预先开发工具、无权限模型 |
| **A2A** | Agent 协作协议 | Agent 通信 | 面向开发者、非终端用户 |
| **AIOS Protocol** | 系统控制协议 | 标准化接口 + 视觉兜底 | 需要适配器生态 |
| **Doubao Phone** | 手机系统 Agent | 视觉 + 系统 hack | 封闭生态、仅限手机 |

### 1.4 AIOS 的独特价值

| 价值 | 说明 |
|------|------|
| 用户导向 | 用户说话 → AI 执行，无需学习命令 |
| 标准化接口 | 使用 D-Bus/API 而非截图点击，更快更可靠 |
| 安全优先 | 5 级权限模型 + 用户确认机制 |
| 渐进式适配 | 自动发现软件 → 基础适配 → 高级适配 |
| 生态兼容 | 可调用 MCP/A2A 扩展能力 |
| **开放协议** | 语言无关、传输灵活、部署多样 |

## 二、三层控制架构

### 2.1 架构总览

```

│                        用户交互层                                │

│        │  端侧小模型      │ ◄───► │   云端大模型          │       │
│        │ (实时/隐私/简单) │       │ (复杂推理/长程规划)    │       │
│        └─────────────────┘       └──────────────────────┘       │
├─────────────────────────────────────────────────────────────────┤
│                     AIOS Protocol 层                            │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────────┐  │
│  │ 权限管理器  │  │ 执行调度器  │  │ 适配器管理器            │  │
│  └─────────────┘  └─────────────┘  └─────────────────────────┘  │
├─────────────────────────────────────────────────────────────────┤
│                       控制层                                     │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────────┐  │
│  │ 系统控制层  │  │  视觉控制层 │  │ 应用控制层              │  │

└─────────────────────────────────────────────────────────────────┘

### 2.2 第一层：系统控制层

**目标**：AI 控制操作系统基础功能，无需安装外部软件

**技术基础**：
| 接口 | 用途 | 技术 |
|------|------|------|
| D-Bus | 系统服务通信 | Session Bus / System Bus |
| gsettings/dconf | 桌面设置 | D-Bus + 配置库 |
| systemd | 服务和电源管理 | D-Bus API |
| NetworkManager | 网络管理 | D-Bus API |
| PulseAudio/PipeWire | 音频控制 | D-Bus + pactl |
| XDG Desktop Portal | 沙箱应用接口 | D-Bus (Wayland 必需) |

**可控制功能**：
| 类别 | 功能 | 权限级别 |
|------|------|---------|
| 桌面设置 | 壁纸、主题、显示设置 | low |
| 音频控制 | 音量、静音、输出设备 | low |
| 电源管理 | 关机、重启、休眠、锁屏 | critical/high |
| 网络设置 | WiFi 连接、VPN | medium |
| 文件管理 | 文件操作、目录管理 | medium/high |
| 系统通知 | 发送通知 | low |

### 2.3 第二层：通用软件控制层

**目标**：AI 监控和操作用户安装的通用软件

**控制方式优先级**：
| 优先级 | 方式 | 说明 | 适用场景 |
|--------|------|------|---------|
| P1 | 原生 API / MCP | 软件提供的编程接口 | 浏览器、办公软件 |
| P2 | AT-SPI / UIA | 系统级无障碍 API | 标准 GUI 应用 |
| P3 | **视觉控制 (Vision)** | **截图 + 视觉模型分析** | **无 API 的应用 (通用兜底)** |
| P4 | GUI 自动化 | 传统脚本/坐标点击 | 最后手段 |

**软件分类**：
| 类别 | 软件 | 控制方式 |
|------|------|---------|
| 浏览器 | Chrome, Firefox, Edge | DevTools Protocol |
| 办公软件 | LibreOffice | UNO API |
| 通讯软件 | 微信, Telegram | AT-SPI / Bot API |
| 媒体软件 | VLC, Spotify | D-Bus MPRIS |
| 文件管理 | Nautilus | D-Bus + AT-SPI |

### 2.4 第三层：专业软件控制层

**目标**：AI 接管专业软件，通过自然语言进行复杂操作

**集成层级**：
| 层级 | 名称 | 实现方式 | 控制深度 | 示例软件 |
|------|------|---------|---------|---------|

| P3 | 命令行 | CLI 接口 | ⭐⭐⭐ | FFmpeg, ImageMagick |
| P4 | GUI 自动化 | AT-SPI + 视觉 | ⭐⭐ | 无 API 的闭源软件 |

**专业软件 API 汇总**：
| 类别 | 软件 | API | 语言 |
|------|------|-----|------|
| 3D/CAD | Blender | bpy | Python |

| 图像处理 | GIMP | Python-Fu | Python |

## 三、混合计算与核心组件

### 3.1 云-端混合架构 (Hybrid-Cloud Architecture)

AIOS 采用**"端侧优先，云端增强"**的混合计算架构，以平衡隐私、延迟和智能：

#### 双栈运行模式 (参考豆包手机技术)

| 层次 | 模式名称 | 模型类型 | 典型任务 | 延迟目标 | 隐私性 |
|------|---------|---------|---------|---------|-------|
| **端侧 (Local)** | Standard Mode | 小模型 (如 Llama 8B, Phi-3) | 意图路由、隐私数据处理、简单指令(打开App、切歌) | < 500ms | 🔒 完全本地 |
| **云端 (Cloud)** | Pro Mode | 大模型 (如 GPT-4, Claude 3.5) | 复杂推理、长程规划、代码生成、跨应用数据整合 | < 2s | ☁️ 需脱敏 |

#### 智能路由策略

```
任务输入 → 复杂度评估
              ↓
    ┌─────────┴─────────┐
    ↓                   ↓
  简单任务            复杂任务
  (打开App、切歌)    (跨应用编排、代码生成)
    ↓                   ↓
  Standard Mode       Pro Mode
  端侧处理              云端处理
  (<500ms)            (<2s)
```

### 3.2 视觉控制层 (Vision Layer)

针对缺乏 API 的应用，AIOS 引入标准化的视觉控制层：

#### 核心组件

| 组件 | 功能 | 技术基础 |
|------|------|---------|
| **通用视觉适配器** | 基于 VLM 分析 UI 语义树 | UI-TARS 类模型 / GPT-4 Vision |
| **仿生输入模拟** | 模拟人类操作轨迹 | 贝塞尔曲线 + 随机延迟 |
| **动态路由器** | 智能选择控制方式 | API → AT-SPI → Vision |

#### 仿生操作算法 (参考豆包手机技术)

> [!TIP]
> 仿生操作是规避应用风控的关键技术

| 参数 | 算法 | 说明 |
|------|------|------|
| **轨迹生成** | 贝塞尔曲线 | 模拟人类手指关节的自然摆动 |
| **压力值** | AXIS_PRESSURE 随机化 | 0.3-0.9 范围内随机 |
| **接触面积** | AXIS_SIZE 随机化 | 0.1-0.3 范围内随机 |
| **操作延迟** | 固定+抖动 | 1000-5000ms 强制延迟 |

#### 动态路由策略

```
用户指令 → 目标应用识别
              ↓
    ┌─────────┴─────────┐
    ↓                   ↓
  有原生API?          无原生API
    ↓                   ↓
  P1: API调用         检查AT-SPI
    ↓                   ↓
  成功/失败         有控件信息?
    ↓                   ↓
  返回/降级          P2: AT-SPI
                        ↓
                     成功/失败
                        ↓
                     P3: Vision
```

```

                                    ↓

| 来源 | 监控方式 | 信息 |
|------|---------|------|

| tool.aios.yaml | 文件检测 | 完整能力声明 |

| L0 | 基础适配 | 启动/关闭 | 自动生成 |
| L1 | CLI 适配 | 命令行参数控制 | 分析 --help |
| L2 | API 适配 | 深度控制 | 第三方开发/官方 |

> **方法命名迁移说明**：旧版 `tool.*` 已迁移至 `capability.*`，`discovery.*` 已迁移至 `registry.*`。

### 5.2 权限模型

| 级别 | 名称 | 用户确认 | 自动授权 | 示例 |
|------|------|---------|---------|------|
| 0 | public | 无需 | ✅ | 读取时间 |
| 1 | low | 首次 | ✅ | 读取设置、调整音量 |
| 2 | medium | 首次 | ⚠️ 可配置 | 打开浏览器、网络请求 |
| 3 | high | 每次 | ❌ | 发送消息、写入文件 |
| 4 | critical | 二次确认 | ❌ | 关机、删除文件 |

### 5.3 沙箱隔离级别

| 级别 | 技术 | 适用场景 | 性能开销 |
|------|------|---------|---------|
| L0 | 无隔离 | 受信任的系统工具 | 无 |
| L1 | 进程隔离 | 一般工具 | 低 |
| L2 | WASM 沙箱 | 第三方工具 | 低 |
| L3 | 容器隔离 | 不可信工具 | 中 |
| L4 | VM 隔离 | 高风险操作 | 高 |

## 六、与 MCP/A2A 的关系

### 6.1 协议定位对比

| 协议 | 定位 | 核心功能 | 目标用户 |
|------|------|---------|---------|
| **AIOS** | AI 控制系统的协议 | 自然语言 → 系统控制 | 终端用户 |
| **MCP** | AI 调用工具的协议 | 工具定义 → AI 调用 | 开发者 |
| **A2A** | AI 之间协作的协议 | Agent 通信 → 任务协作 | 开发者 |

### 6.2 AIOS 调用 MCP/A2A

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIOS Protocol (控制层)                        │
│              用户自然语言 → AI 理解 → 执行控制                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐  │
│  │  原生系统控制    │  │  MCP 调用       │  │  A2A 调用       │  │
│  │  (D-Bus/CLI)    │  │  (扩展工具)     │  │  (远程协作)     │  │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.3 AIOS 与 MCP/A2A 的协作

AIOS 与 MCP 是**互补关系**，而非"安全层"关系：

| 能力 | 说明 |
|------|------|
| **系统控制域权限** | AIOS 的 5 级权限模型专为系统控制设计，是协议核心特性 |
| **桥接策略叠加** | 桥接 MCP 工具时，可叠加本地用户确认、审计、最小权限策略 |
| **统一调用接口** | 自动检测协议类型，无缝切换 |
| **跨协议审计** | 统一的操作日志和追踪 |

> **注意**：MCP 在认证授权方面持续演进（如 OAuth 2.1），其目标是工具调用域。AIOS 的权限模型专注于系统控制域的执行安全。

## 七、技术栈选型

### 7.1 核心技术栈

| 组件 | 技术 | 理由 |
|------|------|------|
| 核心运行时 | Rust | 性能、安全、WASM 支持 |
| 沙箱 | Wasmtime | 默认拒绝权限、性能好 |
| 工作流 | Temporal | 持久化、重试 |
| 策略引擎 | OPA | ABAC 标准 |
| 通信协议 | JSON-RPC 2.0 | 与 MCP 兼容 |
| 配置格式 | YAML | 开发者友好 |
| Schema | JSON Schema | 标准化验证 |

### 7.2 多语言 SDK

AIOS 提供多语言 SDK，开发者可选择熟悉的技术栈：

| 语言 | SDK | 成熟度 | 适用场景 |
|------|-----|--------|---------|
| **Python** | `aios-sdk` | ⭐⭐⭐⭐⭐ | AI集成、快速开发 |
| **TypeScript** | `@aios/sdk` | ⭐⭐⭐⭐⭐ | Web/Node.js生态 |
| **Go** | `aios-go` | ⭐⭐⭐⭐ | 高性能服务 |
| **Rust** | `aios-sdk` (crate) | ⭐⭐⭐⭐ | 系统级控制 |
| **Java/Kotlin** | `aios-jvm` | ⭐⭐⭐ | 企业应用 |
| **C#** | `AIOS.SDK` | ⭐⭐⭐ | .NET生态 |

### 7.3 传输方式支持

| 传输方式 | 协议 | 适用场景 | 性能 |
|---------|------|---------|------|

| **Unix Socket** | Unix Domain Socket | 本地高性能通信 | 极高 |
| **HTTP** | HTTP/1.1, HTTP/2 | 远程访问、云部署 | 中等 |
| **WebSocket** | WS/WSS | 流式响应、实时通知 | 高 |
| **gRPC** | HTTP/2 + Protobuf | 高性能微服务 | 极高 |

| **本地服务** | systemd | 桌面环境、后台服务 |

## 八、生态集成架构

### 8.1 MCP 桥接

AIOS 可以桥接任何 MCP 服务器，为其添加权限控制和安全沙箱：

```
┌─────────────────────────────────────────────────────────────────┐
│                    MCP 桥接架构                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────┐     ┌─────────────────┐                   │
│  │   AIOS Client   │────►│   MCP Bridge    │                   │
│  │   (权限控制)    │     │   (协议转换)     │                   │
│  └─────────────────┘     └────────┬────────┘                   │
│                                   │                             │
│          ┌────────────────────────┼────────────────────────┐   │
│          │                        │                        │   │
│          ▼                        ▼                        ▼   │
│  ┌───────────────┐    ┌───────────────┐    ┌───────────────┐  │
│  │ MCP Server 1  │    │ MCP Server 2  │    │ MCP Server N  │  │
│  │ (filesystem)  │    │ (github)      │    │ (database)    │  │
│  └───────────────┘    └───────────────┘    └───────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**AIOS 为 MCP 添加的能力**：

| 能力 | MCP 原生 | AIOS 增强 |
|------|---------|---------|
| 权限控制 | ❌ | ✅ 5级权限模型 |
| 用户确认 | ❌ | ✅ 敏感操作确认 |
| 沙箱隔离 | ❌ | ✅ 多级沙箱 |
| 审计日志 | 基础 | ✅ 完整追踪 |

### 8.2 OpenAPI 自动转换

现有 REST API 可以自动转换为 AIOS 适配器：

```bash
# 从 OpenAPI 规范生成适配器
aios-cli generate --from openapi --spec ./api.yaml --output ./adapter
```

**自动映射规则**：

| OpenAPI | AIOS |
|---------|------|
| GET endpoint | 只读能力 (risk: low) |
| POST endpoint | 写入能力 (risk: medium) |
| DELETE endpoint | 删除能力 (risk: high) |

### 8.3 远程适配器支持

适配器可以部署在远程，通过 HTTP/WebSocket 访问：

```yaml
# 远程适配器配置
adapters:
  - id: com.example.cloud-tool
    transport: http
    url: "https://my-adapter.example.com/aios"
    auth:
      type: bearer
      token: "${ADAPTER_TOKEN}"
```

- 实现 OpenAPI 自动转换
- 实现 A2A 桥接
- 建立适配器市场
- 开源发布
