**版本**: 2.0.0  
**更新日期**: 2026-01-09

## 一、核心定位

**AIOS Protocol 是让 AI 通过自然语言控制操作系统和软件的开放标准协议。**

```
用户 (自然语言) → AI 引擎 (意图理解) → AIOS Protocol (执行控制) → 操作系统/软件
```

### 1.1 与现有方案的差异

| 方案 | 定位 | 控制方式 | 局限性 |
|------|------|---------|--------|
| **Claude Computer Use** | 桌面自动化 | 截图 + 坐标点击 | 慢、易出错、依赖视觉 |
| **OpenAI Operator** | 浏览器自动化 | GUI 交互 | 仅限浏览器、已停用 |
| **MCP** | 工具调用协议 | API 调用 | 需预先开发工具 |
| **AIOS Protocol** | 系统控制协议 | 标准化接口 | 需要适配器生态 |

### 1.2 AIOS 的独特价值

1. **用户导向**：用户说话 → AI 执行，无需学习命令
2. **标准化接口**：使用 D-Bus/API 而非截图点击，更快更可靠
3. **安全优先**：5 级权限模型 + 用户确认机制
4. **渐进式适配**：自动发现软件 → 基础适配 → 高级适配
5. **生态兼容**：可调用 MCP/A2A 扩展能力

- **Claude Computer Use**：2024年10月发布，Asana、Canva、DoorDash 等公司已在探索
- **OpenAI Operator**：2025年1月发布，8月停用，被 ChatGPT agent 取代
- **自动化趋势**：79% 的 Claude Code 对话涉及自动化任务

### 2.2 用户需求

根据 Anthropic Economic Index 报告：
- 高 GDP 国家用户偏好"增强"模式（协作学习）
- 低 GDP 国家用户偏好"自动化"模式（任务委托）
- 以色列、新加坡、澳大利亚是人均 AI 使用率最高的国家

### 2.3 安全关注

- OWASP 于 2025 年 12 月发布 Agentic Top 10 安全风险清单
- AIOS 专注系统控制域的权限模型，与 MCP/A2A 形成互补

## 三、三层控制架构

### 3.1 第一层：系统控制层

**目标**：AI 控制操作系统基础功能，无需安装外部软件

#### 技术基础

| 接口 | 用途 | 技术 |
|------|------|------|
| **D-Bus** | 系统服务通信 | Session Bus / System Bus |
| **gsettings/dconf** | 桌面设置 | D-Bus + 配置库 |
| **systemd** | 服务和电源管理 | D-Bus API |
| **NetworkManager** | 网络管理 | D-Bus API |
| **PulseAudio/PipeWire** | 音频控制 | D-Bus + pactl |
| **XDG Desktop Portal** | 沙箱应用接口 | D-Bus (Wayland 必需) |

#### 可控制功能

| 类别 | 功能 | 权限级别 |
|------|------|---------|
| **桌面设置** | 壁纸、主题、显示设置 | low |
| **音频控制** | 音量、静音、输出设备 | low |
| **电源管理** | 关机、重启、休眠、锁屏 | critical/high |
| **网络设置** | WiFi 连接、VPN | medium |
| **文件管理** | 文件操作、目录管理 | medium/high |
| **系统通知** | 发送通知 | low |

#### 用户示例

```
用户: "帮我把壁纸换成蓝色的"
AI: 理解意图 → 调用 AIOS → D-Bus 设置壁纸
结果: 壁纸已更换

用户: "10分钟后关机"
AI: 理解意图 → 调用 AIOS → 请求用户确认 → systemd 定时关机
结果: 已设置定时关机

用户: "把屏幕亮度调到50%"
AI: 理解意图 → 调用 AIOS → D-Bus 调整亮度
结果: 亮度已调整
```

#### 与 Claude Computer Use 的对比

| 方面 | Claude Computer Use | AIOS 系统控制 |
|------|-------------------|--------------|
| 控制方式 | 截图 + 坐标点击 | D-Bus API 调用 |
| 速度 | 慢（需要截图分析） | 快（直接 API） |
| 可靠性 | 易出错（UI 变化） | 可靠（标准接口） |
| 安全性 | 依赖模型训练 | 协议层权限控制 |

### 3.2 第二层：通用软件控制层

**目标**：AI 监控和操作用户安装的通用软件

#### 控制方式优先级

| 优先级 | 方式 | 说明 | 适用场景 |
|--------|------|------|---------|
| **P1** | 原生 API | 软件提供的编程接口 | 浏览器、办公软件 |
| **P2** | AT-SPI | Linux 无障碍 API | GUI 应用 |
| **P3** | CLI | 命令行接口 | 支持 CLI 的应用 |
| **P4** | GUI 自动化 | 截图 + 点击 | 最后手段 |

#### 软件分类与控制方案

**浏览器**
| 软件 | 控制方式 | 能力 |
|------|---------|------|
| Chrome | DevTools Protocol | 打开网页、搜索、填表、购物比价 |
| Firefox | Remote Protocol | 同上 |
| Edge | DevTools Protocol | 同上 |

**办公软件**
| 软件 | 控制方式 | 能力 |
|------|---------|------|
| LibreOffice | UNO API | 创建/编辑文档、表格操作 |
| WPS Office | AT-SPI | 基础文档操作 |
| Microsoft Office | COM (Windows) | 完整办公功能 |

**通讯软件**
| 软件 | 控制方式 | 能力 |
|------|---------|------|
| 微信 | AT-SPI / GUI 自动化 | 发送消息、查看消息 |
| Telegram | Bot API + AT-SPI | 消息管理 |
| 邮件客户端 | AT-SPI | 收发邮件 |

**媒体软件**
| 软件 | 控制方式 | 能力 |
|------|---------|------|
| VLC | D-Bus MPRIS | 播放、暂停、切换 |
| Spotify | D-Bus MPRIS | 播放控制 |
| 音乐播放器 | D-Bus MPRIS | 播放控制 |

#### 用户示例

```
用户: "用浏览器帮我比较一下京东和淘宝上这款耳机的价格"
AI: 
  1. 启动 Chrome
  2. 打开京东，搜索耳机，提取价格
  3. 打开淘宝，搜索耳机，提取价格
  4. 比较并返回结果
结果: "京东价格 299 元，淘宝价格 279 元，淘宝更便宜"

用户: "打开 Word 写一份会议纪要"
AI:
  1. 启动 LibreOffice Writer
  2. 创建新文档
  3. 插入会议纪要模板
  4. 等待用户输入内容
结果: 文档已创建，等待编辑

用户: "帮我查看微信里小明发的最新消息"
AI:
  1. 激活微信窗口
  2. 搜索联系人"小明"
  3. 读取最新消息
结果: "小明说：明天下午3点开会"
```

### 3.3 第三层：专业软件控制层

**目标**：AI 接管专业软件，通过自然语言进行复杂操作

#### 集成层级

| 层级 | 名称 | 实现方式 | 控制深度 | 示例软件 |
|------|------|---------|---------|---------|
| **P1** | 原生 API | 软件内置脚本 API | ⭐⭐⭐⭐⭐ | Blender, FreeCAD, Krita |
| **P2** | 插件适配 | 开发 AIOS 插件 | ⭐⭐⭐⭐ | GIMP, Inkscape, LibreOffice |
| **P3** | 命令行 | CLI 接口 | ⭐⭐⭐ | FFmpeg, ImageMagick |
| **P4** | GUI 自动化 | AT-SPI + 视觉 | ⭐⭐ | 无 API 的闭源软件 |

#### 专业软件 API 汇总

**3D/CAD 软件**
| 软件 | API | 语言 | 能力 |
|------|-----|------|------|
| Blender | bpy | Python | 完整场景控制、建模、渲染 |
| FreeCAD | FreeCAD Python | Python | 参数化建模、装配 |
| OpenSCAD | 脚本语言 | OpenSCAD | 程序化建模 |

**图像处理软件**
| 软件 | API | 语言 | 能力 |
|------|-----|------|------|
| GIMP | Python-Fu | Python | 图像编辑、滤镜 |
| Krita | Python 插件 | Python | 绘画、图像处理 |
| Inkscape | inkex | Python | 矢量图形编辑 |

**音视频处理**
| 软件 | API | 语言 | 能力 |
|------|-----|------|------|
| FFmpeg | CLI | 命令行 | 视频转码、剪辑 |
| ImageMagick | CLI | 命令行 | 图像批处理 |
| Audacity | Scripting | Nyquist | 音频编辑 |

#### 用户示例

```
用户: "用 Photoshop 把这张图片的背景去掉"
AI:
  1. 启动 GIMP（Linux 替代）
  2. 打开图片
  3. 使用前景选择工具选中主体
  4. 反选并删除背景
  5. 导出为 PNG
结果: 背景已去除，保存为 output.png

用户: "用 CAD 画一个直径50mm的齿轮"
AI:
  1. 启动 FreeCAD
  2. 创建新文档
  3. 使用 Part 工作台
  4. 调用齿轮生成器
  5. 设置参数：直径=50mm, 齿数=20
结果: 齿轮模型已创建

用户: "用 Blender 创建一个低多边形的小狗模型"
AI:
  1. 启动 Blender
  2. 创建基础网格
  3. 使用 Decimate 修改器降低多边形
  4. 调整形状
  5. 添加材质
结果: 低多边形小狗模型已创建
```

#### 可视化反馈机制

专业软件操作需要用户能看到 AI 的操作过程：

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| **实时操作显示** | 用户看到 AI 的操作过程 | 所有场景 |
| **操作高亮覆盖层** | 高亮 AI 正在操作的元素 | GUI 操作 |
| **步骤面板** | 显示操作步骤列表 | 复杂任务 |
| **操作录制回放** | 操作完成后可回放 | 学习和审查 |

## 四、与 MCP/A2A 的关系

### 4.1 协议定位对比

| 协议 | 定位 | 核心功能 | 目标用户 |
|------|------|---------|---------|
| **AIOS** | AI 控制系统的协议 | 自然语言 → 系统控制 | 终端用户 |
| **MCP** | AI 调用工具的协议 | 工具定义 → AI 调用 | 开发者 |
| **A2A** | AI 之间协作的协议 | Agent 通信 → 任务协作 | 开发者 |

### 4.2 AIOS 调用 MCP/A2A

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIOS Protocol (控制层)                        │
│              用户自然语言 → AI 理解 → 执行控制                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐  │
│  │  原生系统控制    │  │  MCP 调用       │  │  A2A 调用       │  │
│  │  (D-Bus/CLI)    │  │  (扩展工具)     │  │  (远程协作)     │  │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.3 AIOS 的核心价值

1. **系统控制域权限模型**：专注操作系统和桌面应用控制的 5 级权限
2. **桥接时策略叠加**：调用 MCP/A2A 时可叠加本地确认、审计、沙箱策略
3. **统一调用接口**：自动检测协议，无缝切换
4. **审计与追踪**：跨协议操作统一日志

## 五、安全模型

### 5.1 五级权限模型

| 级别 | 名称 | 用户确认 | 自动授权 | 示例 |
|------|------|---------|---------|------|
| 0 | public | 无需 | ✅ | 读取时间 |
| 1 | low | 首次 | ✅ | 读取设置、调整音量 |
| 2 | medium | 首次 | ⚠️ 可配置 | 打开浏览器、网络请求 |
| 3 | high | 每次 | ❌ | 发送消息、写入文件 |
| 4 | critical | 二次确认 | ❌ | 关机、删除文件 |

### 5.2 能力令牌 (Capability Token)

```json
{
  "token_id": "cap-token-xxx",
  "tool_id": "com.example.browser",
  "permission_id": "aios.permission.network.internet.connect",
  "scope": "https://jd.com/*",
  "issued_at": "2026-01-02T10:00:00Z",
  "expires_at": "2026-01-02T11:00:00Z",
  "revocable": true
}
```

### 5.3 沙箱隔离级别

| 级别 | 名称 | 适用场景 | 技术 |
|------|------|---------|------|
| L0 | 无隔离 | 受信任的系统工具 | 直接执行 |
| L1 | 进程隔离 | 一般工具 | 独立进程 |
| L2 | 容器隔离 | 第三方工具 | Docker + gVisor |
| L3 | VM 隔离 | 高风险操作 | Firecracker microVM |

### 5.4 AI Guardrails

- **输入验证**：防止提示注入攻击
- **输出过滤**：防止敏感信息泄露
- **行为监控**：检测异常操作模式
- **速率限制**：防止滥用

## 七、设计原则

1. **用户优先**：一切以用户自然语言交互为核心
2. **安全可控**：敏感操作需要用户确认
3. **渐进增强**：从系统控制到通用软件到专业软件逐步扩展
4. **开放生态**：允许第三方开发软件适配器
5. **标准兼容**：可调用 MCP/A2A 等外部协议
6. **隐私保护**：本地优先，敏感数据不上传
