# AIOS Route B 构思文档

**更新日期**: 2026-03-08

---

> **系统开发口径修订（2026-03-08）**
> AIOS 统一定义为 **AI 原生操作系统 / 系统软件工程**。本文如提及桌面应用、Electron 客户端、应用适配器、App 安装等内容，除非明确标注为“原型期 / 兼容层 / 历史实现”，否则不再代表目标形态。
> 当前最高约束：**系统镜像、系统服务、系统壳层、设备/能力抽象、权限与更新恢复**。

## 1. 为什么必须转向系统开发

如果 AIOS 继续停留在“桌面应用 + 自动化 daemon”路径，它最多只能成为一个高级助手。
只有进入操作系统层，AI 才能真正成为：

- 默认的人机入口
- 系统资源调度参与者
- 能力权限的一等主体
- 会话与工作流的系统级组织者

## 2. AIOS 的系统级假设

- 用户不应该先找 App，再让 AI 帮忙点按钮
- 用户应该先表达意图，再由系统决定调用哪个服务、哪个 shell 能力、哪个兼容提供者
- 系统应该原生理解任务、上下文、风险和恢复路径

## 3. 未来 AIOS 的形态

### 3.1 不是什么
- 不是另一个 Electron 客户端
- 不是一个聚合 MCP 工具的控制面板
- 不是“把现有桌面自动化做得更像 OS”

### 3.2 是什么
- 一个可启动的系统镜像
- 一组长期运行的系统服务
- 一个 AI 优先的系统壳层
- 一个用 capability 和 policy 管理风险的操作环境

## 4. 协议层应该怎么变

协议不再围绕“如何调用 App 功能”展开，而围绕：

- `system.*`：核心系统能力
- `service.*`：第一方系统服务能力
- `shell.*`：会话 / 工作区 / 焦点 / 窗口 / 通知
- `device.*`：硬件与传感器
- `compat.*`：对旧世界的桥接

## 5. 仓库层应该怎么变

需要逐步把仓库拆到这几个工作域：

- `image/`：镜像、启动、恢复、更新
- `services/`：系统服务
- `shell/`：AI Shell / compositor
- `policy/`：能力策略、token、审计
- `compat/`：浏览器、旧应用、SaaS、MCP/A2A 兼容层
- `legacy/`：原型期控制台

## 6. 最重要的概念转变

> **AIOS 的核心问题不再是“怎么做一个更强的系统应用”，而是“如何把 AI 变成系统本身的组织原则”。**
