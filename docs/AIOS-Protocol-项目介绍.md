**更新日期**: 2026-03-08
**定位**: AI 原生操作系统 / 系统软件平台

---

> AIOS 统一定义为 **AI 原生操作系统 / 系统软件工程**。本文如提及桌面应用、Electron 客户端、应用适配器、App 安装等内容，除非明确标注为“原型期 / 兼容层 / 历史实现”，否则不再代表目标形态。

## 一、项目定义

AIOS 不是一个“能控制系统的 AI 应用”。
AIOS 的目标是构建一个 **AI 原生操作系统平台**，使 AI 成为系统默认入口、任务编排器、能力调度者和安全策略参与者。

## 二、我们要解决的问题

现有桌面与移动计算环境存在三个根本问题：

1. 用户必须理解应用边界，而不是表达真实意图
2. 系统能力、服务能力、外部工具能力彼此割裂
3. AI 常被附着在应用表层，无法成为系统级能力

## 三、AIOS 的解决方案

AIOS 用系统级分层来组织 AI：

- 用 **系统镜像** 定义运行边界
- 用 **系统服务** 承载 AI 编排、策略、会话和设备抽象
- 用 **原生壳层** 提供 AI 优先的人机交互
- 用 **compat layer** 兼容旧应用、浏览器和外部生态

## 四、目标产品形态

| 层 | 目标 |
|----|------|
| 镜像层 | 可启动、可回滚、可恢复 |
| 服务层 | agentd / policyd / sessiond / deviced / updated |
| 壳层 | AI launcher、workspace、notification、focus、task board |
| 兼容层 | 浏览器、办公、专业软件、MCP/A2A、SaaS |

## 五、当前仓库角色

当前仓库中的 Electron 客户端和 Node daemon 仍然有价值，但仅限于：

- 验证 AI 编排接口
- 验证 compat provider 设计
- 提炼未来系统服务 API
- 作为 Route B 早期的调试控制台

它们不再代表 AIOS 的最终形态。

## 六、路线图摘要

1. 修正文档和协议口径
2. 建立可启动开发镜像
3. 将编排与策略逻辑系统服务化
4. 构建 AI Shell / compositor
5. 建立权限、审计、更新与恢复闭环

## 七、合作方向

AIOS 优先欢迎以下合作：

- Linux / graphics / compositor 工程
- 系统安全与 capability policy
- 设备抽象与硬件 bring-up
- 本地 AI 推理与系统调度
- 更新、回滚与可靠性工程
