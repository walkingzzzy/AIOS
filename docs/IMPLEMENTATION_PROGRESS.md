# AIOS Route B 实施进展

**更新日期**: 2026-03-08

---

> **系统开发口径修订（2026-03-08）**
> AIOS 统一定义为 **AI 原生操作系统 / 系统软件工程**。本文如提及桌面应用、Electron 客户端、应用适配器、App 安装等内容，除非明确标注为“原型期 / 兼容层 / 历史实现”，否则不再代表目标形态。
> 当前最高约束：**系统镜像、系统服务、系统壳层、设备/能力抽象、权限与更新恢复**。

## 1. 现状结论

当前仓库**不是 AIOS 本体**，而是一个原型期兼容层：

- `packages/client`：Electron 调试控制台
- `packages/daemon`：编排与工具调用原型
- `adapters/*`：用户态兼容提供者
- `MCP/A2A`：外部桥接能力

这些能力证明了 AI 编排与系统控制的方向，但**尚不足以构成“操作系统开发完成度”**。

## 2. 已完成的原型能力

| 能力域 | 状态 | 说明 |
|--------|------|------|
| AI 编排 | ✅ | 三层路由、流式输出、复杂任务规划 |
| 会话持久化 | ✅ | 会话、消息、任务历史 |
| 用户态控制 | ✅ | 文件、浏览器、窗口、系统信息等兼容控制 |
| 桌面控制台 | ✅ | Electron UI、设置、任务板、工具页 |
| 外部桥接 | ✅ | MCP / A2A / WebSocket |

## 3. 仍未开始或未完成的 OS 主线

| OS 工作项 | 状态 | 说明 |
|-----------|------|------|
| 可启动镜像 | ❌ | 仓库尚无系统镜像/启动链产物 |
| PID1 / service manager 集成 | ❌ | 尚未把核心能力重构为系统服务 |
| AI Shell / compositor | ❌ | 当前仍依赖 Electron 而非原生壳层 |
| capability policy | ❌ | 现有权限模型仍偏应用级/adapter 级 |
| 原子更新 / 回滚 / 恢复 | ❌ | 尚无系统级发布机制 |
| 设备抽象与硬件验证 | ❌ | 尚未形成系统级设备栈 |

## 4. 当前代码在 Route B 中的保留价值

### 保留
- Intent / task orchestration
- task lifecycle / streaming model
- session persistence
- compat provider 的经验与接口
- MCP/A2A bridge

### 降级
- Electron 主界面
- “应用适配器”叙事
- 面向 App 安装与打包的发布流程

## 5. 下一阶段目标

1. 文档与命名空间完成系统化修订
2. 建立 `agentd` / `sessiond` / `policyd` 的服务边界
3. 设计镜像、更新和恢复基线
4. 将 Electron 客户端明确定义为 `legacy console`

## 6. 进展汇报方式变更

此后所有进展文档统一按下列结构汇报：

- 镜像与启动
- 系统服务
- 壳层与会话
- 权限与审计
- 更新与恢复
- compat layer

不再以“客户端 UI 功能点数量”作为主进展指标。
