# AIOS Route B 发布路线图

**版本**: 2.1.0
**更新日期**: 2026-03-08
**状态**: 系统开发路线生效

---

> **系统开发口径修订（2026-03-08）**
> AIOS 统一定义为 **AI 原生操作系统 / 系统软件工程**。本文如提及桌面应用、Electron 客户端、应用适配器、App 安装等内容，除非明确标注为“原型期 / 兼容层 / 历史实现”，否则不再代表目标形态。
> 当前最高约束：**系统镜像、系统服务、系统壳层、设备/能力抽象、权限与更新恢复**。

## 1. 路线切换结论

自本版起，AIOS 发布节奏不再以“桌面 App 功能数”作为主线，而以 **操作系统可交付物** 为主线：

- 可启动镜像
- 可管理系统服务
- 可运行 AI Shell
- 可执行能力策略
- 可安全更新与回滚

## 2. 阶段规划

| 阶段 | 时间窗 | 目标 | 退出条件 |
|------|--------|------|---------|
| `R0` | 0-2 周 | 文档、术语、仓库分层切换 | 不再以 App 叙事主导 docs / roadmap |
| `R1` | 2-6 周 | 开发镜像与启动链骨架 | 能在 VM 启动 AIOS 开发镜像 |
| `R2` | 6-10 周 | `agentd` / `sessiond` / `policyd` 骨架 | 三个核心系统服务以系统服务方式运行 |
| `R3` | 10-16 周 | Wayland shell / workspace alpha | 可进入 AIOS 会话并执行基础 shell 能力 |
| `R4` | 16-24 周 | 权限、审计、更新回滚 beta | 有能力授权、日志链路、原子更新回滚 |
| `R5` | 24 周+ | 硬件验证与开发者预览 | 至少一套参考硬件矩阵通过 |

## 3. 当前仓库的交付重定义

| 旧交付物 | 新定义 |
|---------|--------|
| Electron 客户端版本 | 兼容层原型版本 |
| daemon 功能完善 | `agentd prototype` 能力沉淀 |
| 适配器数量增长 | compat provider 覆盖度增长 |
| “发布桌面版” | “发布系统开发者预览” |

## 4. 里程碑细化

### R0：冻结错误方向
- 停止继续扩展“系统应用”叙事
- 文档、命名空间、发布标准切换到系统开发
- 将 `app.*` 统一降级为 `compat.*`

### R1：镜像可启动
- 引入镜像构建链
- 建立 VM / QEMU 开发闭环
- 明确恢复与日志方案

### R2：系统服务化
- `aios-agentd` 承接 AI 编排
- `aios-policyd` 承接授权与风险控制
- `aios-sessiond` 承接用户会话与任务历史

### R3：原生壳层
- 登录会话
- launcher / workspace / notification / focus
- 原生系统入口替代 Electron 控制台

### R4：系统级可靠性
- 原子更新
- 回滚与健康检查
- capability token / 审计日志 / 恢复路径

## 5. 发布门槛

任何“AIOS vNext”发布，至少要满足以下条件中的大多数：

- 可以在参考硬件或 VM 中直接启动
- 核心系统服务以系统服务形式托管，而不是依附桌面 App
- 具备会话、策略、任务、日志四条主链路
- 有系统级升级、回滚、恢复方案
- 壳层可以承载 AI 入口与基础系统交互

## 6. 兼容层策略

过渡期内 `packages/client` 与 `packages/daemon` 仍可继续使用，但仅用于：

- 验证编排逻辑
- 验证 compat provider
- 为未来 `agentd` 提炼接口
- 作为系统开发前期的调试控制台

它们不再是 AIOS 的最终交付物定义。
