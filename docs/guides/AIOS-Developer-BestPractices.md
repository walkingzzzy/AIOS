# AIOS 系统组件最佳实践

**更新日期**: 2026-03-08

---

> **系统开发口径修订（2026-03-08）**
> AIOS 统一定义为 **AI 原生操作系统 / 系统软件工程**。本文如提及桌面应用、Electron 客户端、应用适配器、App 安装等内容，除非明确标注为“原型期 / 兼容层 / 历史实现”，否则不再代表目标形态。
> 当前最高约束：**系统镜像、系统服务、系统壳层、设备/能力抽象、权限与更新恢复**。

## 1. system-first

- 先定义系统边界，再写界面与控制台
- 先定义 capability，再写按钮和页面
- 先定义恢复路径，再开放高风险能力

## 2. 服务化原则

- 每个系统服务保持单一职责
- 用本地 IPC / Unix Socket / D-Bus 优先，不要默认走 HTTP
- 所有服务都要有 health / metrics / logs / trace

## 3. 权限与策略

- capability 必须与策略分离
- compat layer 权限不能等同于 system 权限
- 高风险写操作必须可审计、可撤销、可回滚

## 4. 兼容层纪律

- compat provider 是过渡层，不是产品本体
- 不要因为 compat 方便就改变系统架构方向
- 不要用 GUI 自动化定义协议语义

## 5. 可靠性

- 假设服务会重启、消息会丢失、依赖会失败
- 写清楚幂等性、重试、超时和熔断策略
- 所有关键状态都要能在重启后恢复

## 6. 开发文化

- 任何 PR 都必须说明自己属于 `system` / `service` / `shell` / `device` / `compat`
- 任何新文档都必须避免把 AIOS 写成桌面应用
- 任何新功能都必须回答：它如何提升 AIOS 的系统能力，而不是控制台功能数
