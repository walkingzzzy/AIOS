# AIOS Route B 开发规范

**更新日期**: 2026-03-08

---

> **系统开发口径修订（2026-03-08）**
> AIOS 统一定义为 **AI 原生操作系统 / 系统软件工程**。本文如提及桌面应用、Electron 客户端、应用适配器、App 安装等内容，除非明确标注为“原型期 / 兼容层 / 历史实现”，否则不再代表目标形态。
> 当前最高约束：**系统镜像、系统服务、系统壳层、设备/能力抽象、权限与更新恢复**。

## 1. 适用范围

本规范适用于：

- 系统镜像与启动链
- 系统服务
- 壳层 / compositor
- 策略与审计
- compat layer
- 兼容层原型控制台

## 2. 工作流角色

| 角色 | 关注点 | 主要产物 |
|------|--------|---------|
| 平台工程 | 镜像、启动、更新、恢复 | `image/` |
| 服务工程 | `agentd` / `sessiond` / `policyd` | `services/` |
| 壳层工程 | shell / compositor / workspace | `shell/` |
| 安全工程 | capability policy / 审计 / token | `policy/` |
| 兼容层工程 | 浏览器/旧应用/SaaS 桥接 | `compat/` |
| 原型维护 | 控制台、调试工具、验证链路 | `legacy/` |

## 3. 命名与分层

### 命名空间
- `system.*`
- `service.*`
- `shell.*`
- `device.*`
- `compat.*`
- `professional.*`

### 文件与模块
- 不允许把系统主线代码放进 `client` 语义目录
- 不允许新增以“App 管理”为中心的协议命名
- 旧 `app.*` 视为废弃别名，统一迁移为 `compat.*`

## 4. 依赖纪律

- 系统主线尽量避免依赖 Electron
- 壳层优先使用原生图形栈而非 Web UI 容器
- 本地系统控制优先使用本地 IPC，而非远端 HTTP

## 5. 文档纪律

- 文档必须使用系统开发口径
- 若描述原型期 Electron / daemon，必须显式标注为 `legacy` 或 `compat`
- 所有 roadmap 都要以系统交付物为主，而不是客户端功能点

## 6. 验收问题清单

每个设计或 PR 都必须回答：

1. 它属于哪一层？
2. 它是系统主线还是 compat layer？
3. 它的 capability 与 policy 边界是什么？
4. 失败时如何恢复？
5. 它是否会把 AIOS 拉回“应用开发”路径？
