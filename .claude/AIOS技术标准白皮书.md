## 第一章 摘要

### 1.1 项目概述

AIOS Protocol 是 **AI 系统控制的开放标准协议**，定义 AI 如何描述、调用并安全执行系统控制能力（操作系统/桌面应用/专业软件）。

### 1.2 当前实现状态

AIOS v0.1.0 已于 2026年1月11日 发布，实现状态如下：

| 组件 | 状态 | 技术栈 |
|:-----|:-----|:-------|
| AIOS Daemon | ✅ 已实现 | Node.js + TypeScript |
| Electron 客户端 | ✅ 已实现 | Electron 29 + React 18 |
| 系统适配器 | ✅ 18个已实现 | 跨平台 npm 包 |
| 应用适配器 | ✅ 8个已实现 | OAuth API 集成 |
| 三层 AI 架构 | ✅ 已实现 | gpt-4o-mini / gemini-2.5-flash / claude-sonnet |
| 权限管理器 | ✅ 已实现 | 5级权限模型 |
| 任务编排器 | ✅ 已实现 | ReAct 循环 + Function Calling |
| MCP 集成 | ✅ 已验证 | MCP SDK v1.25.1 |
| JSON-RPC 2.0 | ✅ 已实现 | stdio / WebSocket 传输 |

### 1.3 核心价值主张

1. **标准化接口链路**：以 capability 为最小可调用单元，统一命名、参数、返回结构
2. **安全可控闭环**：五级权限 + 用户确认 + 审计
3. **三层 AI 协调**：Fast（意图识别）→ Vision（屏幕理解）→ Smart（任务规划）
4. **跨平台支持**：Windows / macOS / Linux 统一 API
5. **生态扩展**：适配器插件机制，支持第三方扩展

### 1.4 与 MCP/A2A 的关系

| 协议 | 发布方 | 发布时间 | 核心定位 | AIOS集成状态 |
|:-----|:-------|:---------|:---------|:-------------|
| MCP | Anthropic | 2024年11月 | AI与工具/数据源连接 | ✅ 已验证 |
| A2A | Google | 2025年4月 | Agent间协作 | 📋 Phase 2计划 |
| AIOS | 本项目 | 2026年1月 | 系统控制 | ✅ v0.1.0已发布 |

**AIOS 的差异化价值**：

- 系统控制域的深度适配（窗口管理、进程控制、桌面自动化）
- 更细粒度的权限模型（五级权限 + scope/duration）
- 三层 AI 协调架构（Fast/Vision/Smart）
- 多控制方式动态路由（API→AT-SPI→Vision）

## 第四章 三层 AI 协调架构

### 4.1 三层职责定义

| 层级 | 模型 | 职责 | 响应时间 | 成本 |
|:-----|:-----|:-----|:---------|:-----|
| Fast | gpt-4o-mini | 意图识别、工具调用、简单对话 | <500ms | ~$0.000 |
| Vision | gemini-2.5-flash | 屏幕分析、元素定位、内容识别 | <3s | ~$0.001 |
| Smart | claude-sonnet | 任务规划、任务分解、复杂推理 | <5s | ~$0.002 |

### 4.2 协调模式

**模式 A：简单任务（直接执行）**

用户 → IntentClassifier → 直接工具匹配 → 执行 → 结果  
响应时间: <100ms（不经过 AI）

**模式 B：Fast 层处理**

用户 → Fast (gpt-4o-mini) → Function Calling → 工具执行 → 结果  
响应时间: <500ms

**模式 C：视觉任务（Vision + Fast）**

用户 → Vision (gemini-2.5-flash) → 屏幕分析 → Fast → 执行 → 结果  
响应时间: <3s

**模式 D：复杂任务（三层协调）**

用户 → Smart (claude-sonnet) → 任务规划 → [Vision + Fast]×N → Smart 汇总 → 结果  
响应时间: 5-30s

### 4.3 意图分类器支持的直接匹配模式

- 音量控制：调高音量、静音、取消静音
- 亮度控制：调高亮度、调低亮度
- 应用控制：打开 Chrome、关闭 Safari
- 电源控制：锁屏、休眠、关机、重启
- 截图：截图、截屏
- WiFi：打开 WiFi、关闭 WiFi

## 第六章 通信协议

### 6.1 JSON-RPC 2.0

**传输方式**：
- stdio（主要方式，最跨平台）
- WebSocket（可选，实时通信）

**核心方法**：

| 方法 | 说明 |
|:-----|:-----|
| ping | 健康检查 |
| getVersion | 获取版本 |
| getAdapters | 获取适配器列表 |
| invoke | 调用适配器能力 |
| checkPermission | 检查权限 |
| requestPermission | 请求权限 |
| chat | AI聊天（基础路由）|
| smartChat | 智能聊天（三层协调）|
| chatWithTools | 带工具的聊天 |

### 6.2 消息格式示例

**请求**：
```json
{
  "jsonrpc": "2.0",
  "id": "1",
  "method": "invoke",
  "params": {
    "adapterId": "audio",
    "capability": "set_volume",
    "args": { "level": 50 }
  }
}
```

**响应**：
```json
{
  "jsonrpc": "2.0",
  "id": "1",
  "result": {
    "success": true,
    "volume": 50
  }
}
```

## 第八章 技术创新点与竞争优势

### 8.1 核心创新点

1. **三层 AI 协调架构**
   - Fast 层：本地规则 + 轻量 LLM，响应 <100ms
   - Vision 层：屏幕理解，支持 GUI 自动化
   - Smart 层：任务规划，支持复杂工作流

2. **统一权限模型**
   - 五级权限体系，跨平台一致
   - 能力级别的细粒度控制
   - 审计日志支持

3. **适配器插件机制**
   - 标准化接口定义
   - 热插拔支持
   - 第三方扩展能力

4. **跨平台抽象层**
   - 统一 API，屏蔽平台差异
   - 平台脚本桥接（AppleScript/PowerShell/xdotool）

### 8.2 与竞品对比

| 维度 | MCP | A2A | AIOS |
|:-----|:----|:----|:-----|
| 核心场景 | 工具/数据源调用 | Agent协作 | 系统控制 |
| 权限模型 | 用户同意 | 身份验证 | 五级权限 |
| AI架构 | 单层 | 单层 | 三层协调 |
| 跨平台 | 依赖实现 | 依赖实现 | 原生支持 |
| 生态成熟度 | 高 | 中 | 开发中 |
