# 标准工作流程

## 总原则

- **强制深度思考**：任何时候必须首先使用 sequential-thinking 工具梳理问题
- 不必要的问题不询问用户，必须自动连续执行
- 问题驱动优先于流程驱动
- 追求充分性而非完整性
- 动态调整而非僵化执行

---

## 工具链执行顺序

```
sequential-thinking → shrimp-task-manager → 直接执行
```

- 严格按顺序执行，禁止跳过或调换
- 任一环节失败时，记录原因、补救措施与重新执行结果
- 必要时通过人工流程模拟缺失工具并记录

---

## 标准工作流 6 步骤

| 步骤 | 名称 | 描述 |
|------|------|------|
| 1 | 分析需求 | 理解任务目标和约束 |
| 2 | 获取上下文 | 检索相关代码和文档 |
| 3 | 选择工具 | 确定合适的工具和方法 |
| 4 | 执行任务 | 实施具体工作 |
| 5 | 验证质量 | 运行测试和检查 |
| 6 | 存储知识 | 记录决策和经验 |

---

## 研究-计划-实施 5 阶段

### 阶段1：研究
- 阅读材料、厘清约束
- **禁止编码**

### 阶段2：计划
- 制定详细计划
- 定义成功标准

### 阶段3：实施
- 根据计划执行
- 保持小步提交

### 阶段4：验证
- 运行测试或验证脚本
- 记录结果

### 阶段5：提交
- 准备交付文档
- 提供迁移/回滚方案

---

## 工作流程阶段定义

### 阶段0：需求理解与上下文收集

**快速通道**：简单任务（<30字，单一目标）直接进入上下文收集

**复杂任务**：
1. 使用 sequential-thinking 分析需求
2. 执行强制检索清单（7步）
3. 通过充分性验证（7项检查）
4. 生成上下文摘要

### 阶段1：任务规划

- 使用 sequential-thinking 分析上下文摘要
- 通过 shrimp-task-manager 制定计划与验收契约
- 定义接口规格、边界条件、性能要求、测试标准

### 阶段2：代码执行

- 直接编码（使用 Read/Edit/Write）
- 实时记录到 `operations-log.md`
- 遇到问题时使用 sequential-thinking 分析

### 阶段3：质量验证

- 使用 sequential-thinking 进行深度审查
- 生成评分和建议
- 根据评分决策：≥90分通过 / <80分退回 / 80-89分审阅

## 文件结构规范

所有任务工作文件写入 `.claude/` 目录：

```
<project>/.claude/
├── context-summary-[任务名].md   ← 上下文摘要
├── operations-log.md             ← 决策和操作记录
└── verification-report.md        ← 验证报告
```
