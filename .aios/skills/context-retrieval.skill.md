# 强制上下文检索机制

## 核心原则

**绝对禁止**：在未完成上下文检索和验证的情况下直接编码。违反者立即终止任务。

**原则**：研究先于编码，复用优于创造，一致性优于个人偏好。

---

## 编码前强制检索清单（7项必查）

### 检索强度分级

| 任务类型 | 条件 | 执行范围 |
|---------|------|---------|
| 简单任务 | 单文件、<50行、无依赖 | 步骤1-3，简化验证 |
| 中等任务 | 多文件、<200行、少量依赖 | 完整7步，标准验证 |
| 复杂任务 | 架构级、>200行、复杂依赖 | 完整7步+增强验证 |

---

### □ 步骤1：文件名搜索（必须）

```bash
# 使用 desktop-commander 流式搜索
desktop-commander.start_search searchType="files" pattern="关键词"
```

- **目标**：找到5-10个候选文件
- **记录**：找到X个相关文件，重点关注 [列出文件路径]

### □ 步骤2：内容搜索（必须）

```bash
desktop-commander.start_search searchType="content" pattern="函数名|类名|关键逻辑"
literalSearch=true contextLines=5
```

- **目标**：找到关键实现位置
- **记录**：找到X处实现，重点分析 [file:line, file:line]

### □ 步骤3：阅读相似实现（必须≥3个）

```bash
Read file_path  # 深度阅读至少3个相关文件
```

- **目标**：理解实现模式和设计理由
- **关注点**：
  - 实现模式（工厂/单例/装饰器等）
  - 可复用组件（函数/类/工具）
  - 需注意事项（性能/安全/边界）

### □ 步骤4：开源实现搜索（通用功能必做）

```bash
github.search_code query="具体功能实现" language:"语言" repo:"优质仓库"
```

- **目标**：学习最佳实践和避免常见陷阱
- **触发条件**：通用算法、数据结构、设计模式

### □ 步骤5：官方文档查询（涉及库/框架必做）

```bash
context7 resolve-library-id libraryName="库名"
context7 get-library-docs context7CompatibleLibraryID="库ID" topic="相关主题"
```

- **目标**：避免错误用法，掌握最佳实践
- **优势**：官方文档，token高效，最新版本

### □ 步骤6：测试代码分析（必须）

```bash
desktop-commander.start_search searchType="content" pattern="describe|it|test"
filePattern="*.spec.*|*.test.*"
```

- **目标**：理解测试策略和覆盖标准
- **关注点**：测试框架、断言方式、边界条件、Mock策略

### □ 步骤7：模式提取和分析（必须）

```bash
sequential-thinking  # 分析检索结果，提取项目模式
```

- **输出**：
  - 项目约定：命名规范、文件组织、导入顺序
  - 可复用组件：[组件路径列表]
  - 技术选型：为什么用这个方案？
  - 风险点：并发、边界、性能

## 上下文充分性验证

**必须全部回答"是"且提供具体证据，否则禁止进入编码阶段。**

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 能说出至少3个相似实现的文件路径？ | 列出 file1:line, file2:line, file3:line |
| 2 | 理解项目中这类功能的实现模式？ | 描述模式和理由 |
| 3 | 知道有哪些可复用的工具函数/类？ | 列出具体函数/类名和路径 |
| 4 | 理解命名约定和代码风格？ | 具体说明 |
| 5 | 知道如何测试这个功能？ | 参考测试文件和策略 |
| 6 | 确认没有重复造轮子？ | 检查过的模块/文件 |
| 7 | 理解功能的依赖和集成点？ | 依赖和集成位置 |
